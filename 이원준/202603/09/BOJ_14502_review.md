# 코드 리뷰

**파일**: 이원준/202603/09/BOJ_14502.java

제공해주신 알고리즘 풀이에 대해 꼼꼼히 리뷰해 드리겠습니다. 전반적으로 문제 해결을 위한 핵심 로직을 잘 구현하셨고, 깔끔하게 작동하는 코드입니다.

---

### 1. 시간/공간 복잡도 분석

*   **N, M 범위:** N, M이 최대 8이라는 제약 조건이 이 문제의 풀이 방식을 결정하는 데 매우 중요합니다.

*   **시간 복잡도:**
    *   **초기화 (`main` 메소드):** `originalMap`을 읽고 `blankList`를 채우는 데 `O(N*M)` 시간이 소요됩니다.
    *   **조합 (`comb` 메소드):**
        *   빈 공간의 개수를 `K`라고 할 때, `K`는 최대 `N*M` (8*8 = 64)입니다.
        *   `comb` 메소드는 `K`개의 빈 공간 중에서 3개를 선택하는 조합의 수를 탐색합니다. 이는 `K C 3`입니다.
        *   최악의 경우 (모든 칸이 빈 칸인 경우) `64 C 3 = (64 * 63 * 62) / (3 * 2 * 1) = 41664`가지 조합이 생성됩니다.
    *   **바이러스 확산 (`goVirus` 메소드 - 각 조합당 실행):**
        *   `copyMap` 생성: `O(N*M)`. 2차원 배열의 깊은 복사이므로 각 원소를 순회해야 합니다.
        *   초기 바이러스 위치 큐에 추가: `O(N*M)`.
        *   BFS 탐색: 맵의 모든 칸(`N*M`)과 각 칸에서 나가는 엣지(4방향)를 최대 한 번씩 방문하므로 `O(N*M)`입니다.
        *   안전 영역 계산: `O(N*M)`.
        *   따라서 `goVirus` 메소드는 한 번 실행될 때 `O(N*M)` 시간이 소요됩니다.
    *   **총 시간 복잡도:** `O(N*M) + (K C 3) * O(N*M)`
        *   최악의 경우: `O(64) + 41664 * O(64)`
        *   대략 `41664 * 64`는 약 260만(2.6 * 10^6)번의 연산입니다. 이는 1초 시간 제한 내에 충분히 수행 가능한 수준입니다 (일반적으로 10^8 연산/초 기준).

*   **공간 복잡도:**
    *   `originalMap`: `O(N*M)`
    *   `blankList`: `O(K)`, K <= `N*M`
    *   `newWallArr`: `O(1)` (3개의 고정된 크기)
    *   `copyMap` (goVirus 내): `O(N*M)`
    *   `queue` (goVirus 내): 최악의 경우 모든 칸이 큐에 들어갈 수 있으므로 `O(N*M)`
    *   `comb` 재귀 스택: `O(3)` (깊이 3으로 고정)
    *   **총 공간 복잡도:** `O(N*M)`. N, M이 8일 때 `8*8=64`개의 정수형 배열은 매우 작은 메모리 공간이므로 충분합니다.

**결론:** N, M이 작기 때문에 브루트 포스에 가까운 이 조합 탐색 + BFS 시뮬레이션 방식이 시간/공간 제약 내에서 충분히 효율적입니다.

### 2. 논리 오류 여부

논리 오류는 발견되지 않았습니다.
*   세 개의 벽을 세울 모든 경우의 수를 `comb` 함수를 통해 정확히 탐색합니다.
*   각 경우의 수마다 `originalMap`을 변경하지 않고 `copyMap`을 만들어 바이러스 확산을 시뮬레이션합니다. 이는 매우 중요한 부분이며 정확하게 구현되었습니다.
*   BFS를 사용하여 바이러스가 퍼지는 과정을 올바르게 시뮬레이션합니다.
*   안전 영역을 계산하고 최댓값을 업데이트하는 과정도 정확합니다.

---

### 3. 더 효율적인 방법

주어진 N, M 제약 조건(최대 8)과 문제의 성격(조합 선택 후 시뮬레이션)을 고려할 때, 현재 구현된 방식이 일반적으로 가장 직관적이고 효율적인 방법 중 하나입니다.

*   **근본적인 알고리즘 개선:** 더 효율적인 `N C 3` 조합을 찾는 방법이나 BFS보다 빠른 바이러스 확산 시뮬레이션은 이 문제에서는 존재하지 않습니다. 조합 탐색과 BFS는 해당 부분에서 최적의 접근 방식입니다.
*   **미세 최적화:** 아주 미세한 수준의 최적화는 가능할 수 있지만, 전체적인 복잡도에 큰 영향을 주지는 않을 것입니다. 예를 들어:
    *   `copyMap`을 생성할 때, `System.arraycopy`를 사용할 수 있으나 2차원 배열의 경우 각 행에 대해 별도로 호출해야 하므로 코드의 가독성이 떨어질 수 있습니다 (깊은 복사가 필요한 경우). 현재의 이중 루프 방식도 명확하고 효율적입니다.
    *   `goVirus`에서 `copyMap`을 만들고 바로 3개의 벽을 세우는 대신, 벽을 세운 후 `copyMap`을 만드는 방식으로 순서를 살짝 바꿀 수도 있지만, 큰 차이는 없습니다. 현재 코드는 `originalMap`에서 `copyMap`을 먼저 복사하고 그 위에 벽을 추가하는 방식인데, 이는 `originalMap`의 불변성을 유지하는 좋은 습관입니다.

### 4. 놓친 엣지 케이스

특별히 놓친 엣지 케이스는 없어 보입니다.
*   **`blankList`가 3개 미만인 경우:** 문제 조건 상 항상 3개 이상의 빈 칸이 존재하므로 고려할 필요가 없습니다. 만약 그런 상황이 가능하다면, `comb` 함수가 `depth == 3`에 도달하지 못해 `goVirus`가 호출되지 않을 것이고, `maxPlace`는 `Integer.MIN_VALUE`로 남아있게 됩니다. 이 경우 `main` 함수에서 `blankList.size() < 3`인 경우를 미리 체크하여 예외 처리하거나, `maxPlace`의 초기값을 0으로 설정하는 것을 고려할 수 있습니다. 하지만 이 문제에서는 해당되지 않습니다.
*   **초기 바이러스가 없는 경우:** `goVirus`에서 큐가 비어있는 상태로 시작하며, BFS 루프가 실행되지 않습니다. 그 후 기존의 0만 세어서 `maxPlace`를 업데이트하므로 올바르게 작동합니다.
*   **모든 0이 바이러스에 감염되는 경우:** `currentBlank`가 0이 되고, `maxPlace`가 0으로 업데이트됩니다. 올바르게 작동합니다.
*   **N 또는 M이 1인 경우:** 맵이 1xN 또는 Nx1이 되어도 방향 배열과 경계 조건 검사가 정상적으로 작동하므로 문제가 없습니다.

### 5. 코드 개선점 (비판적이되 건설적으로)

훌륭하게 작성된 코드이지만, 몇 가지 사소한 개선점을 제안합니다.

1.  **Scanner 자원 닫기:** `Scanner` 객체는 사용 후 `close()` 메소드를 호출하여 자원을 반납하는 것이 좋습니다. 특히 콘솔 입력 스트림(System.in)의 경우, 프로그램이 종료될 때 OS가 자동으로 닫아주기는 하지만, 명시적으로 닫는 습관을 들이는 것이 좋습니다. `try-with-resources` 구문을 사용하면 편리합니다.

    ```java
    public static void main(String[] args) {
        try (Scanner sc = new Scanner(System.in)) { // 여기에 추가
            N = sc.nextInt();
            M = sc.nextInt();
            // ... 기존 코드 ...
        } // try 블록 종료 시 자동으로 sc.close() 호출
        System.out.println(maxPlace);
    }
    ```

2.  **클래스 멤버 변수 초기화 위치:**
    `maxPlace`, `blankList` 등의 초기화가 `main` 메소드 안에서 이루어지고 있는데, static 멤버 변수라면 클래스 선언 시 바로 초기화하거나, `main` 메소드 내에서 명시적으로 하는 것 중 하나를 선택할 수 있습니다. 현재 방식도 문제가 없지만, `maxPlace = Integer.MIN_VALUE; blankList = new ArrayList<>();` 등은 멤버 변수 선언 시 바로 할 수도 있습니다.

    ```java
    static int N, M;
    static int[][] originalMap;
    static List<int[]> blankList = new ArrayList<>(); // 여기서 바로 초기화
    static int[][] newWallArr;
    static int maxPlace = Integer.MIN_VALUE; // 여기서 바로 초기화
    static int[] dr = {-1, 1, 0, 0};
    static int[] dc = {0, 0, -1, 1};
    ```
    이렇게 하면 `main` 메소드가 조금 더 간결해질 수 있습니다.

3.  **상수(Constant) 활용:**
    벽(1), 빈 공간(0), 바이러스(2)를 매직 넘버(magic number)로 사용하는 대신 `final static int` 상수로 정의하면 코드의 가독성이 향상되고, 나중에 값 변경이 필요할 때 한 곳만 수정하면 되는 장점이 있습니다.

    ```java
    static final int BLANK = 0;
    static final int WALL = 1;
    static final int VIRUS = 2;
    
    // ...
    if(originalMap[i][j] == BLANK) { /* ... */ }
    // ...
    copyMap[point[0]][point[1]] = WALL;
    // ...
    if(copyMap[i][j] == VIRUS) { /* ... */ }
    // ...
    if(copyMap[nr][nc] == BLANK) {
        copyMap[nr][nc] = VIRUS;
    }
    ```

4.  **좌표 클래스 또는 Record 사용 (선택 사항):**
    현재 `int[] {r, c}` 형태로 좌표를 저장하고 있는데, `int[]`는 타입 안전성이 낮고 의미 전달이 명확하지 않을 수 있습니다. 작은 문제에서는 큰 문제가 없지만, 복잡해지면 좌표를 나타내는 `Point` 클래스나 (Java 16부터) `record`를 정의하여 사용하는 것이 좋습니다.

    ```java
    // 클래스 정의 예시
    static class Point {
        int r, c;
        Point(int r, int c) {
            this.r = r;
            this.c = c;
        }
    }

    // 활용 예시
    blankList.add(new Point(i, j));
    // ...
    Queue<Point> queue = new LinkedList<>();
    queue.add(new Point(i, j));
    // ...
    Point curr = queue.poll();
    int r = curr.r;
    int c = curr.c;
    ```
    이렇게 하면 코드의 가독성과 유지보수성이 향상될 수 있습니다. (경쟁 프로그래밍에서는 `int[]`도 많이 사용됩니다.)

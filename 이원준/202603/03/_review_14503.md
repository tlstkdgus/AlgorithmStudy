### BOJ 14503 로봇 청소기 풀이 리뷰

제공해주신 코드는 BOJ 14503 "로봇 청소기" 문제를 해결하기 위한 자바 코드입니다. 전체적인 흐름은 로봇 청소기의 동작 규칙을 시뮬레이션하는 방식으로 보입니다.

---

#### 1. 시간/공간 복잡도

*   **시간 복잡도:** O(N\*M)
    *   입력 (맵) 읽기: N\*M 칸을 읽으므로 O(N\*M)
    *   `while` 루프: 로봇 청소기가 맵을 이동하며 청소하는 과정입니다. 최악의 경우, 로봇이 청소되지 않은 모든 칸을 방문하고 청소해야 합니다. 각 칸은 한 번만 청소( `arr[r][c] = 2; ans++;` )됩니다. 로봇의 이동 경로는 다소 복잡할 수 있지만, 한 번 청소된 칸은 다시 `0`이 되지 않으므로, 유효한 이동은 최대 `N*M` 번 발생합니다. 각 `while` 루프 내에서는 4방향 탐색과 조건 분기가 상수 시간 (`O(1)`) 안에 이루어집니다. 따라서 전체적으로는 맵의 크기에 비례하는 시간 복잡도, 즉 O(N\*M)으로 볼 수 있습니다. (N, M이 최대 50이므로 2500 정도의 연산으로 충분히 빠릅니다.)
*   **공간 복잡도:** O(N\*M)
    *   `arr` (맵) 배열: N\*M 크기의 2차원 배열을 사용하므로 O(N\*M)
    *   `dx`, `dy` 배열 및 기타 변수: 상수 공간 O(1)
    *   총 공간 복잡도는 O(N\*M) 입니다. (N, M이 최대 50이므로 2500개의 int로 충분합니다.)

**결론:** 시간/공간 복잡도 측면에서는 문제의 제약사항(N, M <= 50)을 고려했을 때 매우 효율적이며 충분합니다.

---

#### 2. 논리 오류 (문제의 지시사항과 코드 비교)

문제의 로봇 청소기 동작 방식은 다음과 같습니다.
1.  현재 칸이 아직 청소되지 않았다면 청소한다.
2.  현재 칸의 주변 4칸 중, 청소되지 않은 빈 칸이 없는 경우:
    a.  바라보는 방향을 유지한 채로 한 칸 후진한다.
    b.  뒤쪽 칸이 벽이라 후진할 수 없다면 작동을 멈춘다.
    c.  뒤쪽 칸이 벽이 아니라면 후진하고 1번으로 돌아간다.
3.  현재 칸의 주변 4칸 중, 청소되지 않은 빈 칸이 있는 경우:
    a.  반시계 방향으로 90도 회전한다.
    b.  바라보는 방향을 기준으로 앞쪽 칸이 청소되지 않은 빈 칸인 경우 한 칸 전진하고 1번으로 돌아간다.
    c.  바라보는 방향을 기준으로 앞쪽 칸이 청소할 칸이 아니라면 2번으로 돌아간다. (즉, 회전만 하고 다시 주변 탐색 및 회전을 시도)

**제공해주신 코드의 주요 논리 오류:**

1.  **`cleaned` 변수의 의미와 사용 방식:**
    *   코드에서 `cleaned` 변수는 "주변 4칸 모두 청소되어 있거나 벽으로 막혀 있어 더 이상 청소할 곳이 없는가?"를 판단합니다. 즉, 주변 4칸 중 `0`이 없으면 `true`가 됩니다.
    *   문제의 2, 3단계 로직은 "주변에 청소할 칸이 없으면 후진", "있으면 반시계 회전 후 전진"으로 크게 나뉘는 것이 아니라, **'왼쪽 방향부터 차례대로'** 인접한 칸을 탐색하여 청소할 칸이 있는지 확인하는 반복적인 과정입니다.
    *   **문제의 3단계(`if(!cleaned)`에 해당하는 부분):**
        *   코드에서는 `if(!cleaned)` 블록에 진입했을 때, 단순히 한 번 반시계 회전 (`d = (d + 3) % 4` 에 해당)한 후, 그 방향으로 바로 한 칸 전진할 수 있으면 전진하고 청소합니다.
        *   **오류:** 문제의 3.c.는 "왼쪽 방향에 청소할 공간이 없다면, 그 방향으로 회전하고 다시 2번으로 돌아간다"고 명시되어 있습니다. 즉, 4번 회전할 때까지 계속 왼쪽 칸을 탐색하고 이동을 시도해야 합니다. 코드에서는 한 번 회전 후 바로 이동을 시도하고, 이동하지 못하면 다시 `while` 루프의 시작으로 돌아가 `cleaned`를 재평가합니다. 이는 문제의 "반시계 방향으로 90도 회전한 후, 그 방향 앞 칸을 확인하고, 아니면 다시 회전"하는 반복적인 로직을 구현하지 못했습니다.

2.  **방향 전환 및 전진 로직의 비효율성 및 오류 가능성:**
    *   `if(!cleaned)` 블록 안의 `if-else if` 문은 `d`에 따라 방향을 바꾸고 전진하는 로직인데, 이 부분이 위에서 설명한 문제의 3단계 로직을 잘못 구현했습니다.
    *   예를 들어, `d=0`일 때 `d=3`으로 바꾸고 `dx[d], dy[d]`를 사용하여 새로운 `cr, cc`를 계산합니다. `arr[cr][cc] == 0`이면 이동합니다. 만약 `arr[cr][cc]`가 0이 아니면 이동하지 않고 `while` 루프 처음으로 돌아갑니다. 이 경우, 로봇은 회전만 한 채로 제자리에 남아있게 됩니다.
    *   문제에서는 이동하지 못하면 다시 반시계 방향으로 회전하고 다시 그 방향 앞 칸을 확인하는 과정을 최대 4번 반복해야 합니다. 현재 코드는 이 반복적인 시도를 하지 않습니다.

3.  **후진 로직의 불필요한 복잡성:**
    *   `if(cleaned)` 블록 안의 후진 로직도 `if-else if` 문으로 `d`에 따라 `dx, dy` 인덱스를 수동으로 지정하고 있습니다. `(d + 2) % 4` 와 같은 간단한 수식으로 후진 방향을 계산할 수 있습니다.
    *   후진 시 `arr[cr][cc] == 1`이면 `break`하는 것은 맞지만, `arr[cr][cc]`가 0이나 2인 경우 `else` 블록에서 단순히 `r = cr; c = cc;`로 위치만 업데이트합니다. 이것은 문제의 지시를 따르는 올바른 동작입니다.

**결론적인 논리 오류:**
문제의 핵심인 "주변 4칸 탐색 및 반시계 회전 시도" 로직이 잘못 구현되어 있습니다. 로봇은 청소되지 않은 칸을 찾기 위해 최대 4번 반시계 회전하며 앞 칸을 확인해야 합니다. 현재 코드는 모든 주변 칸을 한 번에 확인하여 청소할 칸이 있으면 단순히 한 번 회전 후 앞 칸만 확인하고, 아니면 루프를 다시 시작합니다. 이로 인해 로봇이 모든 칸을 청소하지 못하거나, 잘못된 경로로 움직일 가능성이 있습니다.

#### 3. 더 효율적인 방법

현재 로직 자체의 효율성은 문제가 없으나, 문제 지시사항에 맞게 구현하는 것이 중요합니다. 효율성은 알고리즘이 주어진 문제를 얼마나 적은 자원으로 푸는지에 대한 평가이고, 현재 구현된 로직은 문제 요구사항을 제대로 따르지 않고 있으므로 "더 효율적인 방법"을 논하기 전에 **올바른 방법**으로 구현하는 것이 우선입니다.

**올바른 구현을 위한 제안:**

`while` 루프 내부의 로직을 다음과 같이 재구성해야 합니다.

```java
// dx, dy 정의 (상, 우, 하, 좌)
// int[] dx = {-1, 0, 1, 0};
// int[] dy = {0, 1, 0, -1};

while (true) { // 무한 루프. 내부에서 종료 조건을 만났을 때 break
    // 1. 현재 칸 청소
    if (arr[r][c] == 0) {
        arr[r][c] = 2; // 청소 완료 표시 (2)
        ans++;
    }

    boolean movedForward = false; // 3.b.에 따라 전진했는지 여부
    int original_d = d; // 2.a. 와 3.c.의 "현재 방향" 유지를 위해 저장

    // 2. 주변 4칸 탐색 (반시계 회전하며)
    for (int i = 0; i < 4; i++) {
        d = (d + 3) % 4; // 반시계 방향으로 90도 회전 (0->3, 1->0, 2->1, 3->2)
                         // dx, dy 배열의 인덱스와 일치시키기 위함

        int nr = r + dx[d]; // 회전 후 바라보는 앞 칸
        int nc = c + dy[d];

        // 3.b. 바라보는 앞 칸이 청소되지 않은 빈 칸이라면 전진
        if (nr >= 0 && nr < N && nc >= 0 && nc < M && arr[nr][nc] == 0) {
            r = nr;
            c = nc;
            movedForward = true;
            break; // 전진했으므로 1번으로 돌아가기 (while 루프 시작으로)
        }
        // 3.c. 아니라면 회전만 하고 다시 탐색 (for 루프 계속)
    }

    if (movedForward) {
        continue; // 1번으로 돌아간다
    }

    // 여기까지 왔다면 4방향 모두 탐색했지만 청소할 칸이 없었다는 의미 (2단계 진입)
    // 2.a. 바라보는 방향을 유지한 채로 한 칸 후진
    // 이 시점에서 d는 `original_d`와 같을 것입니다. (4번 회전하면 제자리)
    int back_d = (d + 2) % 4; // 현재 바라보는 방향 d에서 2칸 이동 (뒤로)

    int br = r + dx[back_d];
    int bc = c + dy[back_d];

    // 2.b. 뒤쪽 칸이 벽이라 후진할 수 없다면 작동 멈춤
    if (br < 0 || br >= N || bc < 0 || bc >= M || arr[br][bc] == 1) {
        break; // 작동 멈춤 (while 루프 종료)
    }
    // 2.c. 뒤쪽 칸이 벽이 아니라면 후진하고 1번으로 돌아간다
    else {
        r = br;
        c = bc;
        // 방향 d는 유지됨 (문제 조건)
    }
}
System.out.println(ans);
```

이 구조가 문제의 로직을 정확히 구현합니다.

#### 4. 놓친 엣지 케이스

현재 논리 오류가 더 근본적이므로 엣지 케이스를 논하기 어렵지만, 올바른 로직이라면 다음과 같은 엣지 케이스를 고려해야 합니다.

*   **초기 위치가 맵의 가장자리 또는 구석인 경우:** `dx`, `dy`를 사용한 인덱스 접근 시 배열 범위를 벗어나지 않도록 `cr >= 0 && cr < N && cc >= 0 && cc < M` 와 같은 경계 조건 체크가 잘 되어 있어 이 부분은 괜찮습니다.
*   **맵 전체가 벽으로 둘러싸인 경우 (초기 위치 제외):** 로봇은 한 칸 청소 후, 주변 4칸이 모두 벽이므로 후진을 시도하고 결국 벽에 막혀 작동을 멈출 것입니다. (올바른 로직이라면)
*   **맵이 매우 작을 때 (예: 1x1, 1x2 등):** 경계 조건 체크가 있다면 문제없이 처리됩니다.
*   **청소할 칸이 아예 없는 경우 (시작 위치 포함 모든 곳이 벽 또는 이미 청소됨):** `ans`는 0 또는 시작 위치만 청소되고 끝날 것입니다.

현재 코드의 논리적 오류 때문에 이러한 엣지 케이스들이 예상대로 동작하지 않을 수 있습니다. 예를 들어, 청소할 칸이 주변에 분명히 있지만 한 번의 왼쪽 회전으로 찾지 못하면 후진을 시도하게 되어 오동작할 수 있습니다.

#### 5. 코드 개선점

위에 제시된 논리 오류 해결이 가장 중요하며, 이를 전제로 한 코드 개선점은 다음과 같습니다.

1.  **로직 재구성:** 위 3번에서 제시된 "올바른 구현을 위한 제안"에 따라 `while` 루프의 내부 로직을 전면적으로 재구성해야 합니다.
2.  **`d` (방향) 관련 로직 단순화:**
    *   **반시계 회전:** `d = (d + 3) % 4;` (현재 `d`가 0,1,2,3일 때 각각 3,0,1,2로 바뀜)
    *   **후진 방향:** `(d + 2) % 4;` (현재 `d`가 0,1,2,3일 때 각각 2,3,0,1로 바뀜)
    이러한 산술 연산을 사용하여 `if-else if` 체인을 대폭 줄일 수 있습니다.
    예시:
    ```java
    // 후진 시
    int back_d_idx = (d + 2) % 4;
    int br = r + dx[back_d_idx];
    int bc = c + dy[back_d_idx];
    // ...
    ```
3.  **변수명 개선:** `cleaned` 변수명은 "모든 주변 칸이 청소되었음"을 의미하는데, `isNoUncleanedNeighbor` 또는 `allNeighborsCleanedOrWall` 과 같이 좀 더 명확하게 의미를 전달하는 이름으로 바꾸면 좋습니다. (물론, 올바른 로직으로 변경하면 이 변수 자체가 필요 없어질 수도 있습니다.)
4.  **`sc.nextLine()` 사용 주의:**
    *   `sc.nextInt()`는 정수 토큰만 읽고 개행 문자는 버퍼에 남겨둡니다.
    *   `sc.nextLine()`은 개행 문자까지 한 줄을 읽습니다.
    *   현재 코드에서 `sc.nextInt();` 이후 `sc.nextLine();`이 두 번 나옵니다.
        *   `sc.nextInt(); // N`
        *   `sc.nextInt(); // M`
        *   `sc.nextLine(); // 이 시점에서 입력 스트림에 남은 개행 문자를 소비할 것`
        *   `sc.nextInt(); // r`
        *   `sc.nextInt(); // c`
        *   `sc.nextInt(); // d`
        *   `sc.nextLine(); // 이 시점에서 입력 스트림에 남은 개행 문자를 소비할 것`
    *   만약 입력 형태가 각 정수 뒤에 개행이 없고, 한 줄에 여러 정수가 오는 형태라면 불필요한 `sc.nextLine()` 호출이 문제를 일으킬 수 있습니다 (다음 줄의 데이터를 읽어버리거나). 하지만 BOJ 문제에서 일반적으로 정수는 공백 또는 개행으로 구분되므로, `nextInt()` 후 `nextLine()`을 사용하여 남은 개행 문자를 처리하는 것은 안전한 방식일 수 있습니다. 다만, 이 코드에서는 `nextInt()`만 계속 호출하다가 마지막에 `nextInt()`만으로 모든 `arr` 값을 읽기 때문에, `sc.nextLine()`이 필수적이지 않을 수도 있습니다. 입력 포맷에 따라 유무가 결정되므로 확인이 필요합니다.

### 최종 요약

제공해주신 코드는 시간/공간 복잡도 측면에서는 문제 없이 효율적입니다. 그러나, 로봇 청소기가 주변을 탐색하고 이동하며 청소하는 **핵심 논리에서 문제 설명과 다른 부분이 존재**하여 올바른 결과를 도출하지 못할 가능성이 매우 높습니다. 특히 "반시계 방향으로 회전하며 청소할 칸을 찾는 반복적인 과정"이 구현되지 않았습니다.

가장 중요한 개선점은 문제 설명에 따라 `while` 루프 안의 로직을 재구성하는 것입니다. 이 부분을 먼저 수정하신다면 정확한 정답을 얻을 수 있을 것이며, 이후 코드의 가독성과 간결성을 위해 방향 관련 `if-else if` 문을 산술 연산으로 단순화하는 것을 권장합니다.

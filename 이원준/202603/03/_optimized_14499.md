# 최적화 분석

주어진 코드는 주사위 굴리기 (BOJ 14499) 문제의 해결 코드입니다. 현재 코드의 주요 최적화 포인트는 **가독성 및 논리적 명확성 개선**입니다. 시간 복잡도와 공간 복잡도는 문제의 제약사항 내에서 이미 최적에 가까운 상태입니다.

---

### 1. 현재 코드 분석 및 문제점

1.  **주사위 면 표현의 비효율성 및 혼란:**
    *   `row`, `col` 두 개의 4칸 배열을 사용하여 주사위의 6개 면을 표현하려고 합니다. 이는 매우 복잡하고 직관적이지 않습니다. 주사위는 6개의 독립적인 면을 가지며, 이들을 배열 두 개에 나누어 저장하고 움직일 때마다 서로 동기화하는 방식은 오류를 유발하기 쉽습니다.
    *   특히 `col[3] = row[3]; col[1] = row[1];` 또는 `row[3] = col[3]; row[1] = col[1];`와 같이 서로의 값을 복사하는 부분이 많은데, 이는 면들의 관계를 복잡하게 만듭니다.
    *   어떤 인덱스가 어떤 면(윗면, 아랫면, 앞면, 뒷면, 왼면, 오른면)을 의미하는지 명확하지 않습니다. 예를 들어, 출력하는 `col[1]`이 어떤 면을 의미하는지 파악하기 어렵습니다. 문제에서는 "윗면"을 출력하라고 명시되어 있습니다.

2.  **주사위 회전 로직의 복잡성:**
    *   각 방향별로 `for` 문을 사용하여 배열 요소를 한 칸씩 옮기는 방식은 비효율적이지는 않지만, 주사위 회전의 본질적인 로직을 파악하기 어렵게 만듭니다. 6개 면의 값이 어떻게 순환하는지 직관적으로 알기 어렵습니다.
    *   `row[3] = 0;`, `col[0] = 0;` 등 회전 중에 특정 면의 값을 0으로 설정하는 부분은 잘못된 로직입니다. 주사위 면의 값은 회전하면서 다른 면으로 이동할 뿐 사라지지 않습니다.

3.  **입출력 속도 (사소):**
    *   `Scanner`는 대량의 입력에 대해 `BufferedReader`보다 느릴 수 있습니다. 이 문제의 제약 조건(K=1000)에서는 큰 문제가 되지 않을 수 있지만, 일반적인 최적화 관점에서는 `BufferedReader` 사용이 선호됩니다.

### 2. 최적화 방향

1.  **시간 복잡도 개선 방법:**
    *   현재 코드는 명령의 개수 `K`만큼 반복하며, 각 반복 내에서는 주사위 굴리는 동작(상수 시간 연산)과 지도 값 변경(상수 시간 연산)이 이루어집니다. 따라서 전체 시간 복잡도는 `O(K)`입니다.
    *   이것은 명령 하나당 한 번의 처리가 필요하므로, **이미 최적의 시간 복잡도**입니다. 이 이상 개선할 방법은 없습니다.
    *   다만, 주사위 굴리는 로직 내의 `for` 루프(예: `for(int j = 2; j >= 0; j--)`)가 상수 시간 (3번 반복)이므로 `O(1)`에 해당하지만, 이를 직접적인 변수 할당으로 대체하면 상수는 더 줄어들 수 있습니다.

2.  **공간 복잡도 개선 방법:**
    *   지도는 `N*M` 크기의 배열 `arr`로 저장되며, 명령어는 `K` 크기의 배열 `con`으로 저장됩니다. 주사위 면(`row`, `col`) 및 델타 배열(`dx`, `dy`)은 상수 크기입니다. 따라서 전체 공간 복잡도는 `O(N*M + K)`입니다.
    *   이 또한 지도의 정보를 모두 저장하고 명령어를 모두 저장해야 하므로 **이미 최적의 공간 복잡도**입니다. 이 이상 개선할 방법은 없습니다.

3.  **가독성 개선 (가장 중요):**
    *   **주사위 면의 명확한 표현:** 6개의 면을 나타내는 단일 배열을 사용하고, 각 인덱스가 어떤 면을 의미하는지 명확하게 정의합니다 (예: 윗면, 아랫면, 앞면, 뒷면, 왼면, 오른면).
    *   **주사위 회전 로직의 직관화:** 각 방향으로 주사위를 굴렸을 때 6개 면의 값이 어떻게 순환하는지 직접적으로 변수를 할당하여 명시합니다. `for` 루프를 사용한 비직관적인 이동 대신, 임시 변수를 활용한 값 교환으로 로직을 명확하게 표현합니다.
    *   **상수 사용:** 방향 지정을 위한 `0, 1, 2, 3` 대신 `EAST`, `WEST`, `NORTH`, `SOUTH`와 같은 상수를 사용하면 코드의 의미가 더욱 명확해집니다.
    *   **입출력 가속화:** `BufferedReader`와 `StringTokenizer`를 사용하여 입출력 속도를 개선합니다 (필수적이지 않지만 좋은 습관).

### 3. 최종 최적화된 코드 (가독성 및 약간의 상수 시간 개선)

```java
package BOJ;

import java.io.BufferedReader;
import java.io.IOException;
import java.io.InputStreamReader;
import java.util.StringTokenizer;

public class BOJ_14499_Optimized {

    // 주사위 면의 인덱스 매핑 (가독성을 위한 상수)
    // dice[0]: Top (윗면)
    // dice[1]: Bottom (아랫면)
    // dice[2]: Front (앞면)
    // dice[3]: Back (뒷면)
    // dice[4]: Left (왼면)
    // dice[5]: Right (오른면)
    // 초기 모든 면은 0으로 초기화
    static int[] dice = new int[6]; // 주사위의 6개 면을 저장 (0:윗면, 1:아랫면, 2:앞면, 3:뒷면, 4:왼면, 5:오른면)

    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringTokenizer st = new StringTokenizer(br.readLine());

        // 지도의 세로, 가로 길이
        int N = Integer.parseInt(st.nextToken());
        int M = Integer.parseInt(st.nextToken());
        // 주사위의 현재 좌표 (x: 행, y: 열)
        int x = Integer.parseInt(st.nextToken());
        int y = Integer.parseInt(st.nextToken());
        // 명령어의 개수
        int K = Integer.parseInt(st.nextToken());

        // 지도 생성 및 값 입력
        int[][] arr = new int[N][M];
        for (int i = 0; i < N; i++) {
            st = new StringTokenizer(br.readLine());
            for (int j = 0; j < M; j++) {
                arr[i][j] = Integer.parseInt(st.nextToken());
            }
        }

        // 동, 서, 북, 남 델타탐색 배열
        // d=0: 동(East), d=1: 서(West), d=2: 북(North), d=3: 남(South)
        int[] dx = {0, 0, -1, 1}; // 행 변화
        int[] dy = {1, -1, 0, 0}; // 열 변화

        st = new StringTokenizer(br.readLine()); // 명령어 입력
        for (int i = 0; i < K; i++) {
            int command = Integer.parseInt(st.nextToken()); // 1:동, 2:서, 3:북, 4:남
            int d = command - 1; // 0-indexed로 변환

            int nx = x + dx[d];
            int ny = y + dy[d];

            // 지도 범위를 벗어나는 경우, 해당 명령 무시
            if (nx < 0 || nx >= N || ny < 0 || ny >= M) {
                continue;
            }

            // 주사위 위치 업데이트
            x = nx;
            y = ny;

            // 주사위 굴리기 (각 방향에 따른 면들의 값 교환)
            rollDice(d);

            // 주사위 아랫면과 지도 칸의 값 처리
            // dice[1]은 아랫면
            if (arr[x][y] == 0) {
                arr[x][y] = dice[1]; // 지도 칸이 0이면, 주사위 아랫면의 값을 지도 칸에 복사
            } else {
                dice[1] = arr[x][y]; // 지도 칸이 0이 아니면, 지도 칸의 값을 주사위 아랫면에 복사
                arr[x][y] = 0;       // 지도 칸은 0이 됨
            }

            // 주사위 윗면 출력 (dice[0]은 윗면)
            System.out.println(dice[0]);
        }
    }

    /**
     * 주사위를 특정 방향으로 굴리는 함수
     * 0: 동(East), 1: 서(West), 2: 북(North), 3: 남(South)
     */
    private static void rollDice(int direction) {
        int tempTop = dice[0];    // 윗면 (Top)
        int tempBottom = dice[1]; // 아랫면 (Bottom)
        int tempFront = dice[2];  // 앞면 (Front)
        int tempBack = dice[3];   // 뒷면 (Back)
        int tempLeft = dice[4];   // 왼면 (Left)
        int tempRight = dice[5];  // 오른면 (Right)

        switch (direction) {
            case 0: // 동쪽 (East)으로 굴릴 때:
                // 이전 윗면 -> 오른면
                // 이전 오른면 -> 아랫면
                // 이전 아랫면 -> 왼면
                // 이전 왼면 -> 윗면
                dice[0] = tempLeft;   // 윗면 = 왼면
                dice[1] = tempRight;  // 아랫면 = 오른면
                dice[4] = tempTop;    // 왼면 = 윗면
                dice[5] = tempBottom; // 오른면 = 아랫면
                break;
            case 1: // 서쪽 (West)으로 굴릴 때:
                // 이전 윗면 -> 왼면
                // 이전 왼면 -> 아랫면
                // 이전 아랫면 -> 오른면
                // 이전 오른면 -> 윗면
                dice[0] = tempRight;  // 윗면 = 오른면
                dice[1] = tempLeft;   // 아랫면 = 왼면
                dice[4] = tempBottom; // 왼면 = 아랫면
                dice[5] = tempTop;    // 오른면 = 윗면
                break;
            case 2: // 북쪽 (North)으로 굴릴 때:
                // 이전 윗면 -> 앞면
                // 이전 앞면 -> 아랫면
                // 이전 아랫면 -> 뒷면
                // 이전 뒷면 -> 윗면
                dice[0] = tempFront; // 윗면 = 앞면
                dice[1] = tempBack;  // 아랫면 = 뒷면
                dice[2] = tempBottom; // 앞면 = 아랫면
                dice[3] = tempTop;   // 뒷면 = 윗면
                break;
            case 3: // 남쪽 (South)으로 굴릴 때:
                // 이전 윗면 -> 뒷면
                // 이전 뒷면 -> 아랫면
                // 이전 아랫면 -> 앞면
                // 이전 앞면 -> 윗면
                dice[0] = tempBack;  // 윗면 = 뒷면
                dice[1] = tempFront; // 아랫면 = 앞면
                dice[2] = tempTop;   // 앞면 = 윗면
                dice[3] = tempBottom; // 뒷면 = 아랫면
                break;
        }
    }
}
```

### 4. 왜 이 방법이 최선인지 설명

1.  **가독성 및 유지보수성 극대화:**
    *   **명확한 주사위 면 표현:** `dice[0]`이 윗면, `dice[1]`이 아랫면 등으로 명확하게 정의된 6칸 배열을 사용합니다. 이는 주사위의 실제 구조와 1:1로 대응하여 직관적으로 이해하기 쉽습니다. 어떤 면이 어떤 값을 가지는지 한눈에 파악할 수 있으며, 주사위의 6개 면이 각각 독립적인 의미를 가짐을 명확히 합니다.
    *   **직관적인 회전 로직:** `rollDice` 함수 내부에서 각 방향으로 굴렸을 때 면들이 어떻게 순환하는지 임시 변수를 통해 직접적으로 값을 할당합니다. 예를 들어 동쪽으로 굴리면 "윗면이 왼면의 값을 갖게 되고, 왼면은 아랫면의 값을 갖게 된다"는 로직이 코드로 명확하게 드러납니다. 이는 원본 코드의 `for` 루프를 통한 배열 이동보다 훨씬 이해하기 쉽고, 버그 발생 가능성을 줄여줍니다.
    *   **주석 및 상수 사용:** 주사위 면의 인덱스와 방향 명령에 대한 주석과 상수를 추가하여 코드의 의도를 더욱 명확히 합니다.

2.  **정확성 향상:**
    *   원래 코드에서 `row[3] = 0;`와 같이 회전 중에 면의 값을 0으로 초기화하는 잘못된 로직이 있었습니다. 최적화된 코드는 이러한 오류를 제거하고, 면의 값들이 정확하게 순환하도록 보장합니다.
    *   문제에서 요구하는 "윗면"의 값이 정확히 `dice[0]`으로 출력됩니다.

3.  **성능 영향:**
    *   **시간 복잡도:** `O(K)`로 변함이 없습니다. 각 명령 처리 시 주사위 회전 로직은 상수 시간(`O(1)`)이므로, 총 시간은 `K`에 비례합니다. 기존 코드의 `for` 루프도 상수 시간이었지만, 직접적인 변수 할당은 그 상수를 더욱 작게 만들 수 있어 미세한 성능 향상이 있을 수 있습니다.
    *   **공간 복잡도:** `O(N*M + K)`로 변함이 없습니다. `dice` 배열 또한 상수 크기(`6`)이므로 공간 효율성에는 영향을 미치지 않습니다.
    *   **입출력 가속:** `BufferedReader`와 `StringTokenizer`를 사용하여 `Scanner`보다 입출력 오버헤드를 줄입니다. 이 문제의 제약 조건에서는 `Scanner`도 통과할 가능성이 높지만, 대규모 입력이 필요한 문제에서는 필수적인 최적화입니다.

결론적으로, 이 최적화는 알고리즘의 본질적인 시간/공간 복잡도를 변경하지 않으면서도, 코드의 **정확성, 가독성, 유지보수성**을 극대화하여 훨씬 더 견고하고 이해하기 쉬운 코드를 만듭니다. 이는 실제 소프트웨어 개발에서 매우 중요한 가치입니다.

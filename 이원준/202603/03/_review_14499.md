### BOJ 14503 로봇 청소기 풀이 리뷰

**전반적인 평가:**
코드가 전반적으로 문제의 흐름을 따라가려는 노력은 보입니다. 특히 청소되지 않은 구역을 2로 표시하고 `dx`, `dy`를 사용한 델타 탐색은 잘 구현되었습니다. 하지만 로봇 청소기의 핵심 동작 규칙(특히 2번과 3번 규칙) 해석에 있어서 중요한 논리적 오류가 있습니다. 이 부분을 중점적으로 개선해야 합니다.

---

#### 1. 시간/공간 복잡도

*   **공간 복잡도 (Space Complexity):**
    *   `arr` 배열이 `N * M` 크기이므로, `O(N * M)` 입니다.
    *   나머지 변수들은 상수로 간주할 수 있습니다.
    *   따라서 공간 복잡도는 **O(N * M)** 입니다. (N, M 최대 50이므로 50*50 = 2500, 매우 효율적입니다.)

*   **시간 복잡도 (Time Complexity):**
    *   `while` 루프가 로봇이 멈출 때까지 실행됩니다.
    *   루프 내에서:
        *   현재 칸 청소 및 `ans` 증가: `O(1)`
        *   주변 4칸 탐색: 고정된 4번의 반복이므로 `O(1)`
        *   방향 전환 및 이동/후진: `O(1)`
    *   로봇은 각 칸을 최대 한 번만 청소할 수 있습니다. 이미 청소된 칸(2)이나 벽(1)으로는 이동할 수 없거나, 이동하더라도 청소 행위는 발생하지 않습니다. 로봇이 한 번 청소한 칸을 다시 청소하는 일은 없습니다.
    *   후진하거나 방향을 전환하는 과정에서 이미 청소된 칸을 여러 번 지나갈 수 있지만, `N*M` 개의 칸을 넘어서는 무한 루프는 벽에 막히거나 청소할 공간이 없어 멈추는 조건 때문에 발생하지 않습니다.
    *   따라서 전체적으로 모든 칸을 탐색하고 청소하는 데 걸리는 시간은 `N*M`에 비례한다고 볼 수 있습니다.
    *   결과적으로 시간 복잡도는 **O(N * M)** 입니다. (N, M 최대 50이므로 50*50 = 2500, 충분히 효율적입니다.)

#### 2. 논리 오류

가장 중요한 논리 오류는 문제의 로봇 청소기 작동 방식에 대한 해석입니다. 특히 2번, 3번 규칙의 순서와 조건 처리가 문제입니다.

**문제에서 제시된 로봇 청소기 작동 방식 (핵심):**

1.  현재 칸이 아직 청소되지 않았다면 청소한다.
2.  현재 칸의 주변 4칸 중 청소되지 않은 빈 칸이 있는지 탐색한다.
    *   a. 현재 바라보는 방향을 기준으로 왼쪽 방향으로 90도 회전한다.
    *   b. 바라보는 방향을 기준으로 앞쪽 칸이 청소되지 않은 빈 칸인 경우 한 칸 전진하고 1번으로 돌아간다.
    *   c. 청소되지 않은 빈 칸이 없는 경우, 2a로 돌아간다. (총 4번 반복)
3.  2번에서 4방향 모두 탐색했는데도 청소할 칸이 없는 경우 (즉, 4번 회전하고도 갈 곳이 없는 경우):
    *   바라보는 방향을 유지한 채로 한 칸 후진한다.
    *   뒤쪽 칸이 벽이라 후진할 수 없다면 작동을 멈춘다.
    *   뒤쪽 칸이 벽이 아니라면 한 칸 후진하고 1번으로 돌아간다.

**코드의 논리 오류:**

1.  **2번 규칙 (탐색 및 이동) 처리 방식 오류:**
    *   코드는 `cleaned`라는 boolean 변수를 사용하여 주변 4칸에 청소되지 않은 빈 칸이 *있는지 없는지*를 먼저 판단합니다. (`for(int i = 0; i < 4; i++)` 블록)
    *   그리고 `if(cleaned)` (4방향 모두 청소됨/벽)와 `else` (적어도 한 방향은 청소 안 됨)로 분기합니다.
    *   **문제점:** 문제의 2번 규칙은 "왼쪽으로 회전 -> 앞 칸 확인"을 최대 4번 반복하는 *순차적인 탐색*입니다. 코드는 이 순차적인 탐색 로직을 따르지 않고, 한 번에 4방향을 모두 탐색한 후, 만약 `else` 블록(청소 안 된 칸이 있다면)으로 진입하면 **한 번만 왼쪽으로 회전하여 앞 칸을 확인하고 바로 이동**해 버립니다.
    *   만약 왼쪽으로 회전한 앞 칸이 벽이거나 이미 청소된 칸이라면, 로봇은 그 자리에 멈춰 서서 `while` 루프가 다시 시작되고, 다시 `else` 블록으로 들어가 한 번 더 왼쪽으로 회전하고 확인하는 식으로 작동합니다. 이 과정은 문제에서 의도한 "4번 회전하여 갈 곳을 찾는다"와는 다릅니다. 문제에서는 4번의 회전 시도 중에 빈 칸을 찾지 못하면 3번 규칙으로 넘어가야 합니다.
    *   현재 코드는 "4번 회전" 로직이 아니라 "한 번 회전하고 시도, 안 되면 다음 턴에 다시 한 번 회전하고 시도..." 가 됩니다.

2.  **3번 규칙 (후진) 진입 시점 오류:**
    *   코드는 `cleaned`가 `true`일 때 (즉, 처음 4방향 탐색에서 모두 벽이거나 청소된 칸일 때) 후진합니다.
    *   **문제점:** 문제의 3번 규칙은 "2번에서 4방향 모두 탐색했는데도 (즉, 4번 회전했는데도) 청소할 칸이 없는 경우"에 발동합니다. 코드의 `cleaned` 변수는 단지 현재 위치에서 인접한 4칸 중 `0`인 칸이 *하나라도 없으면* `true`가 되므로, 이는 "4번 회전 후에도 갈 곳이 없는 경우"와 다릅니다. 이로 인해 후진하는 조건이 실제 문제 요구사항보다 더 일찍 또는 잘못된 상황에서 발동할 수 있습니다.

#### 3. 더 효율적인/정확한 방법

위에서 지적한 논리 오류를 수정하는 것이 가장 중요합니다. 아래는 문제의 로직을 정확하게 반영한 `while` 루프의 구조입니다.

```java
import java.util.Scanner;

public class BOJ_14503_Corrected {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        int N = sc.nextInt();
        int M = sc.nextInt();
        sc.nextLine();

        int r = sc.nextInt();
        int c = sc.nextInt();
        int d = sc.nextInt(); // 0:북, 1:동, 2:남, 3:서
        sc.nextLine();

        int[][] arr = new int[N][M];
        for (int i = 0; i < N; i++) {
            for (int j = 0; j < M; j++) {
                arr[i][j] = sc.nextInt();
            }
        }

        // 델타 탐색용 - 북, 동, 남, 서 (문제의 d와 매칭)
        int[] dr = {-1, 0, 1, 0}; // r 변화량
        int[] dc = {0, 1, 0, -1}; // c 변화량

        int ans = 0;

        while (true) {
            // 1. 현재 칸이 아직 청소되지 않았다면 청소한다.
            if (arr[r][c] == 0) {
                arr[r][c] = 2; // 청소 완료 표시 (2)
                ans++;
            }

            boolean moved = false; // 2번 규칙에 의해 이동했는지 여부
            
            // 2. 현재 칸의 주변 4칸 중 청소되지 않은 빈 칸이 있는지 탐색한다.
            // (왼쪽으로 90도 회전하며 최대 4번 시도)
            for (int i = 0; i < 4; i++) {
                // a. 현재 바라보는 방향을 기준으로 왼쪽 방향으로 90도 회전한다.
                d = (d + 3) % 4; // 북(0)->서(3), 동(1)->북(0), 남(2)->동(1), 서(3)->남(2)

                // b. 바라보는 방향을 기준으로 앞쪽 칸이 청소되지 않은 빈 칸인 경우 한 칸 전진한다.
                int nr = r + dr[d];
                int nc = c + dc[d];

                // 경계선 안에 있고, 벽이 아니며, 청소되지 않은 빈 칸(0)인 경우
                if (nr >= 0 && nr < N && nc >= 0 && nc < M && arr[nr][nc] == 0) {
                    r = nr;
                    c = nc;
                    moved = true; // 이동했음을 표시
                    break; // 이동했으니 1번 규칙으로 돌아가기 위해 이 반복문 탈출
                }
                // c. 청소되지 않은 빈 칸이 없는 경우, 2a로 돌아간다. (for 루프의 다음 반복으로)
            }

            // 3. 2번에서 4방향 모두 탐색했는데도 청소할 칸이 없는 경우 (moved가 false인 경우)
            if (!moved) {
                // 바라보는 방향을 유지한 채로 한 칸 후진한다.
                // 현재 방향 d의 반대 방향으로 이동
                int back_d = (d + 2) % 4; // 북(0) -> 남(2), 동(1) -> 서(3) 등
                int br = r + dr[back_d];
                int bc = c + dc[back_d];

                // 뒤쪽 칸이 벽(1)이라 후진할 수 없다면 작동을 멈춘다.
                if (br < 0 || br >= N || bc < 0 || bc >= M || arr[br][bc] == 1) {
                    break; // 로봇 작동 중지
                } else {
                    // 뒤쪽 칸이 벽이 아니라면 한 칸 후진하고 1번으로 돌아간다.
                    r = br;
                    c = bc;
                    // 방향 d는 그대로 유지된다.
                }
            }
        }

        System.out.println(ans);
        sc.close(); // Scanner 닫기
    }
}
```

**주요 개선 사항:**
*   **로직 일치:** 문제의 2단계와 3단계 규칙을 정확히 순서대로 처리하도록 `while` 루프 내부를 재구성했습니다.
*   **방향 전환:** `d = (d + 3) % 4;`를 사용하여 왼쪽 90도 회전을 간결하게 표현했습니다. `(d + 2) % 4`는 후진 방향을 나타냅니다.
*   **`moved` 플래그:** 2번 규칙에 따라 이동했는지 여부를 명확히 하여 3번 규칙(후진) 진입 시점을 정확하게 제어합니다.

#### 4. 놓친 엣지 케이스

현재 코드의 논리 오류로 인해 다음 엣지 케이스들이 제대로 처리되지 않을 수 있습니다.

*   **주변 4칸 모두 벽이거나 청소되었지만, 왼쪽 방향으로 돌면 청소할 칸이 있는 경우:**
    *   현재 코드에서는 `cleaned`가 `true`가 되어 바로 후진 로직으로 넘어갑니다. 이는 실제 문제의 "왼쪽으로 회전하며 탐색" 규칙을 무시하는 것입니다.
*   **로봇이 갇힌 경우:**
    *   로봇이 벽이나 이미 청소된 칸으로 둘러싸여 있고, 후진할 곳마저 벽인 경우 (예: 코너에 갇힘). 이 경우 로봇은 멈춰야 합니다.
    *   수정된 코드에서는 `break` 조건 (`arr[br][bc] == 1`)이 정확히 이를 처리합니다.
*   **초기 위치가 0이 아닌 경우:**
    *   문제 조건상 로봇은 항상 0인 칸에서 시작하는 것으로 보입니다. 만약 1(벽)에서 시작한다면 `arr[r][c] == 0` 조건 때문에 아무것도 청소하지 못하고 바로 주변 탐색을 시작할 것입니다. 2(청소됨)에서 시작하면 역시 청소하지 않고 주변 탐색. 이는 문제가 요구하는 방식이 아닐 수 있지만, 일반적으로 시작 위치는 유효한 청소 구역으로 주어집니다.

#### 5. 코드 개선점

*   **논리 흐름 단순화:** 위에서 제시된 "더 효율적인/정확한 방법" 섹션의 코드를 참고하여 `while` 루프의 논리 흐름을 개선하는 것이 가장 시급합니다.
    *   `if-else if`로 나열된 방향별 후진 로직 (`d == 0` 일 때 `dx[2], dy[2]`)을 `back_d = (d + 2) % 4;`와 같이 일반화하여 사용할 수 있습니다.
    *   마찬가지로 방향 전환 로직도 `d = (d + 3) % 4;`로 일반화할 수 있습니다. 이렇게 하면 코드의 중복을 줄이고 가독성을 높일 수 있습니다.
*   **상수 사용:** 마법의 숫자(Magic Number)를 상수로 정의하면 코드의 가독성과 유지보수성이 향상됩니다.
    *   `ROOM_EMPTY = 0;`
    *   `ROOM_WALL = 1;`
    *   `ROOM_CLEANED = 2;`
    *   `DIR_NORTH = 0;`
    *   `DIR_EAST = 1;`
    *   `DIR_SOUTH = 2;`
    *   `DIR_WEST = 3;`
*   **Scanner 리소스 관리:** `sc.close();`를 `main` 메서드 끝에 추가하여 `Scanner` 객체가 사용한 시스템 리소스를 해제하는 것이 좋습니다.
*   **`sc.nextLine()` 사용:** `nextInt()` 다음에 `nextLine()`을 사용하여 버퍼에 남아있는 개행 문자를 소비하는 것은 좋습니다. 현재 코드에서는 적절하게 사용되고 있습니다.

**결론:**
제공해주신 코드는 BOJ 14503 문제의 기본적인 델타 탐색 및 맵 관리 방식은 잘 구현했지만, 로봇 청소기의 핵심적인 작동 규칙(특히 주변 탐색 및 이동, 후진 조건)에 대한 해석과 구현에서 논리적 오류가 있습니다. 이 부분을 문제의 규칙에 맞춰 재구성한다면 성공적으로 문제를 해결할 수 있을 것입니다. 위에서 제시된 "더 효율적인/정확한 방법"을 참고하여 코드를 수정해 보시는 것을 강력히 권합니다.

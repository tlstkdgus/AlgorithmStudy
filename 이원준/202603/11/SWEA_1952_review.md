### **SWEA_1952 알고리즘 풀이 리뷰**

전반적으로 문제의 요구사항을 잘 이해하고 재귀적인 방식으로 답을 찾아가는 **백트래킹 (Backtracking) 또는 깊이 우선 탐색 (DFS)** 접근 방식을 잘 구현하셨습니다. 가지치기(Pruning) 로직도 적용되어 있어 불필요한 탐색을 줄이려는 노력이 보입니다.

---

### **1. 시간/공간 복잡도**

*   **시간 복잡도:**
    *   `swimfee` 함수는 각 달(`m`)에 대해 최대 3가지 선택지(1일권, 1달권, 3달권)를 탐색합니다.
    *   재귀의 깊이는 최대 12 (1월부터 12월까지)입니다.
    *   가장 비효율적인 최악의 경우, 가지치기가 거의 발생하지 않는다면, 대략 `O(3^N)`의 시간 복잡도를 가집니다. 여기서 `N`은 개월 수 (12개월)입니다.
    *   `3^12 = 531,441` 입니다. 이 정도 연산량은 `N=12`인 경우 일반적인 시간 제한(보통 1초) 내에 통과할 수 있습니다. (테스트 케이스의 수에 따라 달라질 수 있지만, 단일 테스트 케이스 기준으로는 충분합니다.)
    *   하지만 가지치기 (`if(fee >= ans) return;`) 덕분에 실제 연산 수는 훨씬 줄어들 것입니다.
*   **공간 복잡도:**
    *   재귀 호출 스택의 깊이가 최대 12~14 (12개월 + 3달권 때문에 최대 12+3-1 = 14까지 `m`이 올라갈 수 있음)이므로 `O(N)`의 공간 복잡도를 가집니다. 여기서 `N`은 개월 수입니다. (12개월은 상수이므로 `O(1)`로 볼 수도 있습니다.)
    *   `month` 배열은 13칸으로 `O(1)` 공간입니다.
    *   따라서 공간 복잡도는 매우 효율적입니다.

---

### **2. 논리 오류 여부**

제공해주신 코드에는 **심각한 논리 오류는 없습니다.**
몇 가지 중요한 포인트들을 정확하게 처리하고 있습니다:

*   **초기 `ans` 설정:** 1년권 가격 `y`로 `ans`를 초기화하여, 1년권 자체가 최소 비용이 될 수 있거나, 다른 조합의 비용 상한선 역할을 할 수 있도록 한 것은 올바른 접근입니다.
*   **가지치기 (Pruning):** `if(fee >= ans) return;` 조건은 현재까지 누적된 비용이 이미 찾아낸 최소 비용(`ans`)보다 크거나 같다면 더 이상 탐색할 필요가 없음을 의미하며, 이는 백트래킹의 핵심적인 최적화 기법입니다.
*   **기저 조건 (`m > 12`):** 모든 달에 대한 처리가 끝나면 `ans`를 업데이트하고 재귀를 종료하는 것은 올바릅니다.
*   **사용하지 않는 달 처리:** `if(month[m] == 0)` 부분에서 해당 달에 이용 계획이 없다면 비용을 추가하지 않고 다음 달로 넘어가는 것은 정확합니다.
*   **3개월권 처리 (`swimfee(m + 3, fee + m3)`):** `m + 3`으로 다음 달을 건너뛰는 것은 3개월권이 현재 달부터 3개월 동안 유효함을 잘 반영합니다. 만약 `m + 3`이 12월을 넘어가는 경우(`m`이 11월, 12월일 때), `m > 12` 기저 조건에 의해 자연스럽게 처리됩니다. 예를 들어, 11월에 3개월권을 사용하면 11, 12월이 커버되고, 다음 재귀 호출은 `swimfee(14, fee+m3)`가 되어 바로 기저 조건에 도달, 종료됩니다.

### **3. 더 효율적인 방법**

현재 코드는 백트래킹 방식이며, 동일한 "상태" (예: 특정 달부터 시작하는 최소 비용)를 여러 번 계산할 수 있습니다. 이 문제는 **동적 계획법 (Dynamic Programming, DP)**으로 훨씬 더 효율적으로 해결할 수 있습니다.

**DP 접근 방식:**

*   **`dp[i]` 정의:** `dp[i]`를 `i`월부터 12월까지 수영장을 이용하는 데 필요한 **최소 비용**이라고 정의합니다.
*   **점화식:** 각 `i`월에 대해 4가지 선택지 중 최소값을 취합니다.
    *   **1일권:** `dp[i+1] + (month[i] * d)`
    *   **1달권:** `dp[i+1] + m1`
    *   **3달권:** `dp[i+3] + m3`
    *   **1년권:** `y` (1년권은 전체 기간에 대한 비용이므로, 최종 결과에서만 고려하거나, `dp[1]`을 구할 때 최종적으로 `y`와 비교하는 방식으로 처리할 수 있습니다. 여기서는 `dp[i]`를 12월까지의 최소 비용으로 정의했으므로, 1월부터 시작하는 `dp[1]`의 결과와 `y`를 최종적으로 비교하는 것이 더 자연스럽습니다.)

*   **계산 순서:** `dp` 배열은 12월부터 거꾸로 1월까지 계산합니다. (또는 1월부터 12월까지 계산할 수도 있습니다. 이 경우 `dp[i]`를 `1월부터 i월까지의 최소 비용`으로 정의하고, `i-1`, `i-3` 등의 값을 참조하게 됩니다. 여기서는 일반적으로 더 직관적인 `i`월부터 `12`월까지의 최소 비용으로 정의하겠습니다.)
    *   `dp[13] = 0` (12월 이후에는 비용이 0)
    *   `dp[14] = 0`
    *   `dp[15] = 0` (3달권을 처리하기 위함)
*   **예시 (12월부터 역순):**
    *   `dp[12]` = `min(`
        *   `dp[13] + (month[12] * d)`,
        *   `dp[13] + m1`,
        *   `dp[15] + m3` (이 경우 `dp[15]`는 0이므로 `m3`만 더해짐)
        `)`
    *   `dp[11]` = `min(`
        *   `dp[12] + (month[11] * d)`,
        *   `dp[12] + m1`,
        *   `dp[14] + m3`
        `)`
    *   ...
    *   `dp[1]` = `min(`
        *   `dp[2] + (month[1] * d)`,
        *   `dp[2] + m1`,
        *   `dp[4] + m3`
        `)`

*   **최종 결과:** `min(dp[1], y)`

**DP의 복잡도:**
*   **시간 복잡도:** `N`개의 달 각각에 대해 상수 시간 연산 (3가지 비교)을 수행하므로 `O(N)`입니다. `N=12`이므로 매우 빠릅니다.
*   **공간 복잡도:** `dp` 배열 저장에 `O(N)` (15칸) 공간이 필요합니다.

이 DP 방식은 현재의 백트래킹 방식보다 **시간 복잡도 측면에서 훨씬 더 효율적**입니다.

### **4. 놓친 엣지 케이스**

현재 구현은 대부분의 엣지 케이스를 잘 처리하고 있습니다:

*   **모든 달에 이용 계획이 없는 경우:** `month` 배열의 모든 값이 0인 경우, `ans`는 `y`로 초기화된 후, 재귀 호출에서 `fee`가 계속 0으로 유지되어 `ans`는 `Math.min(y, 0)`이 됩니다. 이는 올바르게 0 또는 `y` 중 작은 값을 반환합니다.
*   **1년권이 가장 싼 경우:** `ans`가 `y`로 초기화되어 있으므로, 다른 어떤 조합도 `y`보다 싸지 않다면 `y`가 최종 답이 됩니다. 가지치기 덕분에 불필요한 탐색도 줄어듭니다.
*   **마지막 달(12월)에 3개월권 선택:** 12월에 3개월권을 선택하면 비용 `m3`가 추가되고 `swimfee(15, fee + m3)`가 호출됩니다. `15 > 12` 이므로 기저 조건에 의해 `ans`가 `Math.min(ans, fee + m3)`로 업데이트되고 종료됩니다. 이는 3개월권이 남은 기간(12월)만 커버하더라도 전체 3개월권 비용을 지불하는 상황을 올바르게 반영합니다.
*   **월별 이용 일수가 0인데 1일권/1달권/3달권을 선택하는 경우 (논리적 오류는 아니지만 비효율적):**
    *   예를 들어 `month[m] == 0` 인데, `m1`이 0이 아니라면 `swimfee(m + 1, fee + m1)`를 선택할 수도 있습니다. 하지만 `month[m] == 0` 일 때는 `swimfee(m + 1, fee)`만 호출하는 현재 로직이 가장 효율적이며 정확합니다. 즉, 현재 코드는 이 부분을 놓치지 않고 잘 처리하고 있습니다.

### **5. 코드 개선점**

1.  **동적 계획법 (DP)으로 전환:**
    *   가장 큰 개선점입니다. 위 3번에서 설명한 DP 방식으로 전환하면 `O(3^N)`에서 `O(N)`으로 시간 복잡도를 혁신적으로 개선할 수 있습니다.

2.  **상수 정의:**
    *   `13` (`month` 배열 크기), `12` (최대 월), `1` (최소 월) 같은 '매직 넘버'들을 `static final int` 상수로 정의하면 코드의 가독성과 유지보수성이 향상됩니다.
        ```java
        static final int MAX_MONTH = 12;
        static final int MONTH_ARRAY_SIZE = MAX_MONTH + 1; // 1-indexed
        // ...
        month = new int[MONTH_ARRAY_SIZE];
        for(int i = 1; i <= MAX_MONTH; i++) { // <= MAX_MONTH
            month[i] = sc.nextInt();
        }
        // ...
        if(m > MAX_MONTH) { // base case
        // ...
        }
        ```

3.  **`Scanner` 자원 해제:**
    *   실제 애플리케이션에서는 `Scanner` 객체 사용 후 `sc.close()`를 호출하여 자원을 해제하는 것이 좋습니다. 경쟁 프로그래밍에서는 일반적으로 생략되지만, 좋은 습관입니다. `System.in`은 프로그램 종료 시 자동으로 닫히므로 필수까지는 아닙니다.

4.  **전역 변수 (Global Variables) 사용:**
    *   `d, m1, m3, y, month, ans`를 모두 전역 변수로 사용하고 있습니다. 경쟁 프로그래밍에서는 자주 사용되는 방식이지만, 일반적인 소프트웨어 개발에서는 함수에 필요한 파라미터만 전달하거나, 클래스 멤버 변수로 캡슐화하는 것이 권장됩니다. 이 문제의 규모에서는 크게 문제가 되지 않습니다.

### **결론**

제공해주신 코드는 백트래킹 방식으로 문제의 해답을 정확하게 찾아내며, 가지치기를 통해 어느 정도 최적화도 이루어져 있습니다. 이 정도 규모의 문제(N=12)에서는 충분히 통과할 수 있는 솔루션입니다.

하지만 더 효율적인 방법을 찾는다면, **동적 계획법(DP)**으로 전환하는 것이 가장 큰 성능 개선을 가져올 것입니다. 이는 알고리즘 문제 해결 능력을 한 단계 더 발전시키는 좋은 연습이 될 것입니다.

### 1. 시간/공간 복잡도

*   **시간 복잡도:**
    *   입력 처리: 맵의 크기 `N*M` (16x16)에 비례합니다. 각 테스트 케이스마다 `O(N*M)`.
    *   DFS: 최악의 경우, DFS는 모든 칸을 한 번씩 방문하게 됩니다. 각 칸을 방문할 때마다 상하좌우 4방향을 탐색하므로, `O(N*M)`의 시간이 소요됩니다.
    *   총 시간 복잡도: 테스트 케이스당 `O(N*M)`입니다. 10개의 테스트 케이스가 있으므로 총 `10 * O(N*M)`.
    *   `N=16`이므로 `16*16 = 256`입니다. 이는 매우 작은 값이므로, 현재의 구현으로도 시간 제한에는 전혀 문제가 없습니다. `O(256)`은 사실상 상수 시간에 가깝습니다.

*   **공간 복잡도:**
    *   `map` 배열: `O(N*M)` (16x16 int 배열).
    *   `visited` 배열: `O(N*M)` (16x16 boolean 배열).
    *   재귀 스택: DFS는 최악의 경우 (미로가 한 줄로 길게 연결된 경우), `N*M` 깊이까지 재귀 호출이 발생할 수 있습니다. 따라서 `O(N*M)`.
    *   총 공간 복잡도: `O(N*M)`입니다. 이것 역시 `O(256)`으로 매우 효율적인 수준입니다.

### 2. 논리 오류

현재 코드에는 심각한 논리 오류는 없지만, 효율성 측면에서 개선할 수 있는 부분이 있습니다.

*   **`isExist` 플래그의 사용과 DFS의 조기 종료:**
    *   `dfs` 함수 내에서 `map[x][y] == 3`을 만나 `isExist = true; return;`을 실행하면 현재 DFS 호출은 종료됩니다.
    *   하지만 이 `return`은 *현재 스택 프레임만* 종료시키는 것이고, 그 상위 호출 스택에서 `for` 루프는 계속해서 다른 방향을 탐색하려고 시도합니다. `isExist`가 `true`가 되었음에도 불구하고, 이미 목적지를 찾았다는 사실을 모르고 불필요한 탐색을 계속 진행하게 됩니다.
    *   **예시:** `2`에서 `3`으로 가는 길이 여러 개 있다고 가정합니다. 첫 번째 경로를 찾으면 `isExist`가 `true`가 되지만, DFS는 여전히 다른 경로들을 계속 탐색하려고 시도하여 자원 낭비를 초래합니다.

    이를 해결하기 위한 방법은 아래 "더 효율적인 방법" 섹션에서 설명하겠습니다.

### 3. 더 효율적인 방법

현재 DFS 구현 자체는 효율적이지만, 위에서 언급한 `isExist` 플래그의 문제 때문에 불필요한 탐색을 줄이는 것이 핵심입니다.

*   **DFS에서 조기 종료 로직 추가:**
    `dfs` 함수를 `boolean` 타입으로 만들어서 경로 발견 여부를 반환하도록 변경하거나, `dfs` 시작 시점에 `isExist`를 체크하여 이미 발견되었다면 바로 종료하도록 할 수 있습니다.

    **방법 A: `dfs` 함수를 `boolean`으로 변경 (권장)**
    ```java
    // static boolean isExist; // 이제 이 플래그는 필요 없음
    static boolean dfs(int x, int y) {
        // 이미 방문한 곳이라면 더 이상 탐색하지 않음 (방문 체크가 여기에서 이루어지도록)
        if (visited[x][y]) return false; // 이 위치는 탐색 불가능 or 이미 탐색함

        visited[x][y] = true; // 현재 위치 방문 처리

        if (map[x][y] == 3) { // 목적지에 도달
            return true; // 경로 발견
        }

        for (int d = 0; d < 4; d++) {
            int cx = x + dx[d];
            int cy = y + dy[d];

            // 경계 및 벽 체크
            if (cx < 0 || cx >= 16 || cy < 0 || cy >= 16 || map[cx][cy] == 1) continue;

            // 다음 탐색으로 경로를 찾으면 바로 true 반환하여 모든 재귀 종료
            if (dfs(cx, cy)) {
                return true;
            }
        }
        return false; // 이 경로로는 목적지에 도달할 수 없음
    }

    // main 함수에서 호출 시:
    // isExist = dfs(startX, startY); // isExist를 지역 변수로 사용하거나 바로 출력
    // System.out.println("#" + tNum + " " + (dfs(startX, startY) ? 1 : 0));
    ```
    이렇게 변경하면, 한 번 목적지를 찾으면 모든 재귀 호출이 `true`를 반환하며 즉시 종료되므로 불필요한 탐색을 완벽하게 막을 수 있습니다.

    **방법 B: `isExist` 플래그를 함수 시작에 체크**
    ```java
    static void dfs(int x, int y) {
        if (isExist) return; // 이미 목적지 발견했다면 더 이상 탐색하지 않음

        visited[x][y] = true;
        if (map[x][y] == 3) {
            isExist = true;
            return;
        }

        for (int d = 0; d < 4; d++) {
            int cx = x + dx[d];
            int cy = y + dy[d];

            if (cx < 0 || cx >= 16 || cy < 0 || cy >= 16) continue;
            if (map[cx][cy] == 1 || visited[cx][cy]) continue;

            dfs(cx, cy);
            if (isExist) return; // 이 부분도 추가해야 불필요한 재귀 호출이 줄어듭니다.
                                 // 하지만 A 방법이 더 깔끔하고 확실합니다.
        }
    }
    ```
    이 방법은 `return` 문을 추가함으로써 `dfs`를 호출한 직후 `isExist`를 확인하여 상위 스택에서도 조기 종료를 시킬 수 있습니다. 하지만 `boolean` 반환 값을 사용하는 방법 A가 더 일반적이고 명확합니다.

*   **BFS (Breadth-First Search) 사용:**
    미로 찾기처럼 '어떤 경로든 존재하는가?' 또는 '최단 경로를 찾는가?' 와 같은 문제는 BFS로도 효과적으로 해결할 수 있습니다. BFS는 큐(Queue)를 사용하여 구현되며, 모든 노드를 레벨별로 탐색하므로 최단 경로를 찾는 데 적합합니다. 여기서는 최단 경로가 필요 없지만, 재귀 대신 반복문으로 구현되어 스택 오버플로우 위험이 없고, 일반적으로 "최초 발견" 시점에 빠르게 종료할 수 있습니다.
    16x16 맵에서는 DFS의 재귀 깊이가 깊지 않아 스택 오버플로우 걱정은 적습니다. 성능 상의 큰 차이는 없겠지만, 다른 관점의 풀이로 고려해볼 수 있습니다.

### 4. 놓친 엣지 케이스

현재 코드는 대부분의 엣지 케이스를 잘 처리합니다.

*   **시작 지점(2)이 바로 도착 지점(3)인 경우:**
    `startX`, `startY`에서 `dfs`가 호출되면 `map[startX][startY]`가 `3`일 경우 `isExist`가 `true`로 설정되고 바로 `return`하므로 올바르게 처리됩니다. (실제 문제에서는 2와 3이 같은 위치에 있지 않을 가능성이 높지만, 일반적인 미로찾기 엣지 케이스로서는 그렇습니다.)
*   **경로가 없는 경우:**
    모든 가능한 경로를 탐색했지만 `3`에 도달하지 못하면 `isExist`는 `false`로 유지되고 `0`이 출력되므로 올바르게 처리됩니다.
*   **시작/도착 지점이 맵의 경계에 있는 경우:**
    `cx < 0 || cx >= 16 || cy < 0 || cy >= 16` 조건으로 경계 바깥으로 나가는 것을 방지하므로 올바르게 처리됩니다.
*   **미로 전체가 벽으로 막혀있는 경우:**
    `map[cx][cy] == 1` 조건으로 벽을 통과하지 않으므로 올바르게 처리됩니다.

SWEA 문제의 특성상 `2`와 `3`은 항상 맵 안에 존재한다고 가정할 수 있습니다.

### 5. 코드 개선점

*   **`isExist` 플래그의 논리적 오류 수정 (가장 중요):** 3번 섹션에서 설명한 DFS 조기 종료 로직을 적용하세요. `dfs` 함수가 `boolean`을 반환하도록 변경하는 것을 강력히 추천합니다.
*   **전역 변수 관리:**
    *   `tNum`: `main` 메서드의 `for` 루프에서 `tc` 변수로 이미 존재하므로 굳이 전역 변수로 선언할 필요가 없습니다. 지역 변수로 `tc`를 사용하는 것이 더 깔끔합니다.
    *   `finishX`, `finishY`: 경로 존재 여부만 판단한다면, 굳이 목적지의 좌표를 전역 변수로 저장할 필요는 없습니다. `dfs` 함수 내에서 `map[x][y] == 3`만으로 충분합니다.
    *   `map`, `visited`: 매 테스트 케이스마다 `new` 연산으로 초기화하고 있기 때문에 전역 변수로 두어도 문제가 되지는 않습니다. 하지만 이런 큰 배열은 함수의 인자로 전달하는 것이 더 일반적인 프로그래밍 습관입니다. 다만, DFS 같은 재귀 함수에서는 인자 전달이 복잡해질 수 있어 전역으로 두는 경우가 많습니다.
    *   `dx`, `dy`: `static final`로 선언하여 상수로 사용하는 것이 좋습니다. `static final int[] dx = {-1, 0, 1, 0};`
*   **매직 넘버 제거:**
    `16`이라는 숫자가 여러 번 사용되고 있습니다. 이를 `static final int SIZE = 16;`와 같이 상수로 선언하여 사용하는 것이 코드의 가독성과 유지보수성을 높입니다. 맵 크기가 변경될 경우 상수 하나만 변경하면 됩니다.

    ```java
    // 상단에 추가
    static final int SIZE = 16;
    // 이후 코드에서 16 대신 SIZE 사용
    // map = new int[SIZE][SIZE];
    // for(int i = 0; i < SIZE; i++) { ... }
    // if(cx < 0 || cx >= SIZE || cy < 0 || cy >= SIZE) continue;
    ```
*   **코드 컨벤션:**
    Java에서는 보통 클래스 멤버 변수(필드)는 `private` 또는 `protected`로 선언하고, 필요한 경우 getter/setter 메서드를 통해 접근하는 것이 권장됩니다. 경쟁 프로그래밍에서는 `public static`으로 많이 사용되지만, 일반적인 관점에서는 알아두면 좋습니다.

### 결론

전반적으로 DFS의 기본 아이디어를 잘 구현하셨습니다. 16x16이라는 작은 맵 크기 덕분에 현재 코드의 효율성 문제(불필요한 탐색)가 시간 제한에 영향을 주지는 않겠지만, 더 큰 맵에서는 문제가 될 수 있습니다. 위에 제시된 `dfs` 함수의 `boolean` 반환값 활용을 통해 조기 종료 로직을 추가하는 것이 가장 중요한 개선점입니다. 이를 통해 더 견고하고 효율적인 코드를 만들 수 있습니다.

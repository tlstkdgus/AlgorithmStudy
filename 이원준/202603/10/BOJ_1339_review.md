### **전반적인 평가:**

알고리즘의 핵심 아이디어인 **각 알파벳이 전체 단어 합에 기여하는 가중치를 계산하고, 이 가중치가 높은 알파벳에 9부터 내림차순으로 숫자를 할당하여 최대 합을 구하는 그리디(Greedy) 방식**은 이 문제의 정석적인 풀이이며 매우 훌륭합니다. 가중치 계산 부분은 정확하게 구현하셨습니다.

하지만 숫자를 할당하고 총합을 계산하는 마지막 루프에서 논리적인 오류가 발견됩니다. 이 부분을 수정하면 완벽한 풀이가 될 것입니다.

---

### **1. 시간/공간 복잡도**

*   **시간 복잡도:**
    *   **입력 및 가중치 계산:** `N`개의 단어를 읽고 각 단어의 길이를 `L_i`라고 할 때, 각 단어의 알파벳을 순회하며 가중치를 계산합니다. 최악의 경우 N=10, L=10이므로, 이 부분은 `O(N * L_max)` (여기서 `L_max`는 가장 긴 단어의 길이) 입니다. `Math.pow`는 내부적으로 루프를 돌지만, 지수가 10 이하로 매우 작으므로 상수 시간 연산으로 볼 수 있습니다.
    *   **정렬:** `alphabet` 배열의 크기는 26으로 고정되어 있습니다. `Arrays.sort(alphabet)`은 `O(K log K)` (여기서 `K=26`) 이므로, 사실상 상수 시간 `O(1)` 입니다.
    *   **결과 계산:** 정렬된 `alphabet` 배열을 순회합니다. 이 역시 `O(K)` (K=26) 이므로 상수 시간 `O(1)` 입니다.
    *   **총 시간 복잡도:** `O(N * L_max)` 입니다. 문제의 제약 조건(N <= 10, L_max <= 10)을 고려하면 `10 * 10 = 100`회 정도의 연산으로 매우 빠릅니다.

*   **공간 복잡도:**
    *   `alphabet` 배열: 크기 26인 `int` 배열이므로 `O(K)` (`K=26`), 즉 상수 공간 `O(1)` 입니다.
    *   `word` 변수: 가장 긴 단어의 길이에 비례하는 공간 `O(L_max)`를 사용하지만, 이는 단어를 읽을 때마다 재사용되므로 전체적으로 상수 공간으로 볼 수 있습니다.
    *   **총 공간 복잡도:** `O(1)` 입니다.

### **2. 논리 오류**

**치명적인 논리 오류가 숫자를 할당하고 총합을 계산하는 마지막 부분에 있습니다.**

```java
		int num = 9;
		while(num > 0) { // 이 조건이 문제
			for(int i = alphabet.length - 1; i >= 0; i--) {
				ans += num*alphabet[i];
				num--;
			}
		}
```

*   **문제점:**
    1.  `num` 변수는 `for` 루프 내부에서 `alphabet.length` (26)번 반복될 때마다 감소합니다. `num`이 9에서 시작하므로, 첫 번째 `for` 루프가 끝날 때 `num`은 `9 - 26 = -17`이 됩니다.
    2.  `num`이 `-17`이 되면 `while(num > 0)` 조건은 즉시 `false`가 되어 `while` 루프는 **단 한 번만 실행**됩니다.
    3.  결과적으로, `alphabet` 배열의 가장 큰 가중치에는 9가 곱해지고, 두 번째에는 8이, 세 번째에는 7이 곱해지는 식으로 진행되다가, **가중치가 0인 알파벳에도 불필요하게 숫자가 할당되고 `num`이 계속 감소하여, 실제로 의미 있는 가중치를 가진 알파벳에 0 이하의 숫자가 곱해질 수 있습니다.**
        *   예를 들어, 만약 고유한 알파벳이 10개라면, `num`은 9부터 0까지 할당되어야 합니다. 하지만 이 코드는 `num`이 9부터 시작해서 26번 감소시키므로, 가장 큰 가중치 10개에 `9, 8, ..., 0`이 할당되고, 나머지 16개 (가중치 0)에는 `-1, -2, ..., -16`이 할당됩니다. 이로 인해 최종 `ans` 값이 틀려집니다.
    4.  정확히는, `num`은 `9`부터 시작해서 `alphabet[25]` (가장 큰 가중치)에 9, `alphabet[24]`에 8, ... 이런 식으로 `alphabet[0]`에 `9 - 25 = -16`을 곱하게 됩니다. 이 방식으로는 올바른 최댓값을 구할 수 없습니다. 고유한 알파벳의 개수만큼만 `num`을 감소시키고 할당해야 합니다.

### **3. 더 효율적인 방법**

현재 채택하신 그리디 전략(가중치 계산 후 정렬, 큰 숫자부터 할당)은 이 문제에 대한 **가장 효율적이고 올바른 접근 방식**입니다. 이보다 더 근본적으로 효율적인 알고리즘은 없습니다.

몇 가지 마이크로 최적화가 가능하지만, 전체적인 시간 복잡도에 큰 영향을 주지는 않습니다.

*   **`Math.pow` 대신 정수 연산:** `Math.pow`는 `double`을 반환하므로 캐스팅이 필요하고 부동소수점 연산이 수반됩니다. `10`의 거듭제곱은 간단한 정수 곱셈으로 대체할 수 있습니다.
    ```java
    // 예를 들어, 각 단어 처리 시
    String word = sc.next();
    int len = word.length();
    int powerOf10 = 1; // 10^0
    for(int j = len - 1; j >= 0; j--) { // 단어를 뒤에서부터 처리
        int idx = word.charAt(j) - 'A';
        alphabet[idx] += powerOf10;
        powerOf10 *= 10;
    }
    // 혹은 현재 코드와 같이 앞에서부터 처리하려면
    // int powerOf10 = (int) Math.pow(10, len - 1);
    // for(int j = 0; j < len; j++) {
    //     int idx = word.charAt(j) - 'A';
    //     alphabet[idx] += powerOf10;
    //     powerOf10 /= 10;
    // }
    ```
    하지만 현재 코드의 `Math.pow`도 작은 지수에 대해서는 성능 저하가 미미합니다.

### **4. 놓친 엣지 케이스**

논리 오류가 있는 부분을 제외하고는 대부분의 엣지 케이스를 잘 처리합니다.

*   **고유한 알파벳의 개수:**
    *   **0개:** (가능성은 희박하지만) 모든 가중치가 0이 되어 `ans`는 0이 됩니다.
    *   **1개:** (예: "AAA") 해당 알파벳 가중치가 커지고 9가 곱해집니다. `num`은 한번만 감소하고 나머지는 0이므로 올바른 결과를 얻습니다. (단, 현재 코드의 버그로 인해 `9 * weight + 8 * 0 + ... + (-16) * 0`이 되어버립니다.)
    *   **10개 이하:** 9부터 차례대로 할당되어야 합니다.
    *   **10개 초과:** 문제의 조건상 10개의 고유한 숫자(0-9)에 10개 이상의 알파벳을 1:1로 매핑할 수는 없습니다. 보통 이런 문제는 고유한 알파벳의 개수가 10개 이하로 주어지거나, 0부터 9까지의 숫자를 다 사용한 후 남은 알파벳은 가중치가 0으로 처리된다는 암묵적인 규칙이 있습니다. 현재의 가중치 기반 그리디는 이 규칙에 부합합니다. (수정된 코드에서는 10개 초과하더라도 9~0까지의 숫자만 사용하게 됩니다.)
*   **단어 길이가 1인 경우:** (예: "A", "B") `Math.pow(10, 0)`이 1이 되어 올바르게 가중치를 계산합니다.
*   **모든 단어의 가중치가 0인 경우:** (예: 빈 단어만 입력되거나 N=0인 경우) `ans`는 0이 됩니다.

### **5. 코드 개선점**

가장 중요한 개선점은 **숫자 할당 및 총합 계산 부분의 논리 오류를 수정**하는 것입니다.

**수정 제안 코드:**

```java
package backtracking;

import java.util.Arrays;
// import java.util.Collections; // Collections는 List를 다룰 때 사용됩니다. 배열 정렬에는 필요 없습니다.
import java.util.Scanner;

public class BOJ_1339 {
	public static void main(String[] args) {
		Scanner sc = new Scanner(System.in);
		int N = sc.nextInt();
		
		int ans = 0;
		
		int[] alphabet = new int[26]; // 각 알파벳의 가중치를 저장

		for(int i = 0; i < N; i++) {
			String word = sc.next();
			int len = word.length();
			
			// 각 알파벳의 자릿수별 가중치를 합산
			// 예: GCA -> G: 100, C: 10, A: 1
			for(int j = 0; j < len; j++) {
				int idx = word.charAt(j) - 'A';
				// Math.pow는 double을 반환하므로 int로 캐스팅
				// 10의 거듭제곱을 직접 계산하는 것이 더 효율적일 수 있습니다.
				alphabet[idx] += (int) Math.pow(10, len - 1 - j);
			}
		}
		
		// 가중치가 높은 알파벳에 큰 숫자를 할당하기 위해 오름차순 정렬
		// 가장 큰 가중치는 배열의 마지막에 위치하게 됨
		Arrays.sort(alphabet);
		
		int currentDigit = 9; // 할당할 가장 큰 숫자 (9부터 시작)
		
		// 정렬된 배열의 뒤에서부터 (가장 큰 가중치부터) 순회하며 숫자 할당
		// 가중치가 0인 경우는 해당 알파벳이 사용되지 않았으므로 숫자를 할당할 필요 없음
		// 고유한 알파벳의 최대 개수는 10개이므로, 0까지 할당하고 나면 더 이상 할당할 숫자가 없으므로 루프를 멈춤
		for(int i = alphabet.length - 1; i >= 0; i--) {
			if (alphabet[i] == 0) {
				// 가중치가 0이면, 이보다 작은 인덱스의 값들도 0일 것이므로 더 이상 계산할 필요 없음
				break; 
			}
			
			// 현재 알파벳의 가중치에 할당할 숫자를 곱하여 총합에 더함
			ans += currentDigit * alphabet[i];
			
			// 다음으로 큰 가중치에는 한 단계 작은 숫자를 할당
			currentDigit--;
            
            // 0까지 모든 숫자를 할당했으면 더 이상 진행할 필요 없음
            if (currentDigit < 0) { 
                break;
            }
		}
		
		System.out.println(ans);
		sc.close(); // Scanner 리소스 해제
	}
}

```

**주요 수정 내용:**

1.  **숫자 할당 루프 수정:** `while(num > 0)` 루프를 제거하고, `for` 루프 내에서 `currentDigit` (이전의 `num`)이 감소하도록 했습니다.
2.  **가중치 0인 경우 처리:** `if (alphabet[i] == 0)` 조건을 추가하여 가중치가 0인 알파벳에는 숫자를 할당하지 않고, `break`를 통해 불필요한 반복을 중단시켰습니다. 정렬된 상태이므로 `alphabet[i]`가 0이면 그보다 작은 인덱스의 값들도 모두 0일 것입니다.
3.  **`Collections` import 제거:** `Collections`는 `List` 컬렉션에 대한 유틸리티 클래스입니다. 배열 정렬에는 `Arrays`만 필요하므로 불필요한 import는 제거했습니다.
4.  **`Scanner` 리소스 해제:** `sc.close()`를 추가하여 사용한 `Scanner` 객체의 자원을 해제했습니다. (필수적이지는 않지만 좋은 습관입니다.)

# 최적화 분석

주어진 코드는 백준 1253번 "좋다" 문제를 해결하는 알고리즘으로 보입니다. 특정 수가 배열 안의 다른 두 수의 합으로 표현될 수 있는지 찾는 문제입니다. 현재 코드는 정렬 후 투 포인터(Two-Pointer) 기법을 사용하여 O(N^2) 시간 복잡도로 동작합니다.

이 문제의 제약사항(N <= 2000)을 고려할 때, O(N^2) 시간 복잡도는 일반적으로 허용되는 수준입니다. 따라서 이 코드의 최적화는 주로 현재 알고리즘의 효율성을 극대화하고, 가독성을 높이는 방향으로 진행됩니다.

---

### 1. 시간 복잡도 개선 방법

**현재 시간 복잡도 분석:**
*   **배열 정렬:** `Arrays.sort(numArr)`은 O(N log N)의 시간 복잡도를 가집니다.
*   **외부 루프:** `for(int i = 0; i < N; i++)`는 N번 반복합니다. 각 반복마다 `numArr[i]`를 타겟으로 설정합니다.
*   **내부 투 포인터 루프:** `while(left < right)`는 정렬된 배열에서 두 수의 합을 찾는 전형적인 투 포인터 기법입니다. 최악의 경우 O(N)의 시간 복잡도를 가집니다.
    *   여기서 `left == i` 또는 `right == i`인 경우를 건너뛰는 조건이 포함되어 있습니다. 이는 타겟 자신을 합산에 사용하지 않기 위함입니다.
*   **총 시간 복잡도:** O(N log N + N * N) = **O(N^2)**

**개선 방향:**
이 문제에서 O(N^2)보다 더 좋은 일반적인 시간 복잡도(예: O(N log N) 또는 O(N))를 달성하기는 어렵습니다.
*   N개의 각 요소가 "좋은 수"인지 확인해야 하므로, 외부 루프(N번 반복)는 필수적입니다.
*   정렬된 배열에서 특정 합을 찾는 투 포인터 기법은 O(N)으로 최적입니다.
*   따라서, O(N * N) = O(N^2)이 이 문제에서 투 포인터를 사용하는 접근 방식으로는 사실상 최적의 시간 복잡도입니다.

**현재 코드에서 이미 최적화된 부분:**
*   `Arrays.sort()`: 투 포인터를 사용하기 위한 필수적인 전처리 과정입니다.
*   `sum == target`일 때 `count++` 후 `break;`: 한 번이라도 타겟을 만들 수 있는 조합을 찾으면 더 이상 탐색할 필요가 없으므로 다음 타겟으로 넘어가는 것은 중요한 최적화입니다.

**추가적인 시간 복잡도 개선은 미미합니다.** 현재 구현 방식이 이 문제의 요구사항을 고려할 때 합리적인 선택입니다.

### 2. 공간 복잡도 개선 방법

**현재 공간 복잡도 분석:**
*   `numArr` 배열: O(N) 공간을 사용합니다.
*   `Scanner` 및 기타 변수: 무시할 수 있는 상수 공간을 사용합니다.
*   `Arrays.sort()`: Java의 `Arrays.sort()`는 기본적으로 Timsort를 사용하며, 이는 최악의 경우 O(N) 또는 평균 O(log N)의 추가 공간을 사용합니다. (원시 타입 배열의 경우 Dual-Pivot Quicksort를 사용하며, 이는 O(log N)의 스택 공간을 사용합니다.)
*   **총 공간 복잡도:** **O(N)**

**개선 방향:**
*   입력 배열(`numArr`)을 저장해야 하므로 O(N) 공간은 필수적입니다.
*   정렬도 인플레이스(in-place)에 가깝게 이루어지므로, 추가적인 공간 복잡도 개선의 여지는 거의 없습니다.
*   따라서 현재 O(N) 공간 복잡도는 이 문제에서 최적이라고 볼 수 있습니다.

### 3. 가독성 개선

현재 코드는 전반적으로 가독성이 좋습니다. 변수명도 명확하고 로직도 이해하기 쉽습니다. 몇 가지 사소한 부분에서 개선할 수 있습니다.

**개선 내용:**
*   **주석 추가:** 각 주요 로직 블록에 대한 설명을 추가하여 코드의 목적을 더 명확히 합니다.
*   **변수명 명확화:** `target`은 `currentNumber` 또는 `targetValue`와 같이 명확히 할 수 있습니다. (현재 `target`도 충분히 좋습니다)
*   **상수 사용:** 만약 N의 범위가 바뀌거나 다른 상수가 있다면 상수를 정의하여 사용하는 것이 좋습니다. (이 문제에서는 해당 없음)
*   **코드 블록 간격:** 논리적인 블록 사이에 빈 줄을 추가하여 시각적으로 구분합니다.

### 4. 최종 최적화된 코드

주어진 문제의 특성상 알고리즘의 근본적인 시간/공간 복잡도를 더 줄이기는 어렵습니다. 따라서 현재 O(N^2) 접근 방식을 유지하되, 가독성과 미세한 효율성을 높인 코드를 제공합니다.

```java
package backtracking; // 패키지명은 문제에 따라 변경될 수 있습니다.

import java.util.Arrays;
import java.util.Scanner;

public class BOJ_1253_Optimized { // 클래스명 변경
	
	public static void main(String[] args) {
		Scanner sc = new Scanner(System.in);
		int N = sc.nextInt();
		
		int[] numArr = new int[N];
		for(int i = 0; i < N; i++) {
			numArr[i] = sc.nextInt();
		}
		
		sc.close(); // Scanner 자원 해제
		
		// 1. 배열 정렬: 투 포인터 기법을 사용하기 위해 필수적입니다. O(N log N)
		Arrays.sort(numArr);
		
		int goodNumberCount = 0; // "좋은 수"의 개수를 저장할 변수명 변경 (count -> goodNumberCount)
		
		// 2. 각 숫자를 '타겟'으로 설정하여 다른 두 수의 합으로 표현 가능한지 확인. O(N)
		for(int i = 0; i < N; i++) {
			int targetValue = numArr[i]; // 현재 타겟 숫자
			
			int left = 0;
			int right = N - 1;
			
			// 3. 투 포인터 탐색: 정렬된 배열에서 targetValue를 만들 수 있는 두 수를 찾습니다. O(N)
			while(left < right) {
				// 타겟 숫자(numArr[i])가 left 포인터와 같으면, 자기 자신을 사용하는 것이므로 건너뜁니다.
				if(left == i) {
					left++;
					continue;
				}
				
				// 타겟 숫자(numArr[i])가 right 포인터와 같으면, 자기 자신을 사용하는 것이므로 건너뜁니다.
				if(right == i) {
					right--;
					continue;
				}
				
				int currentSum = numArr[left] + numArr[right]; // 두 포인터가 가리키는 값의 합
				
				if(currentSum == targetValue) {
					// 합이 타겟과 같으면, numArr[i]는 "좋은 수"입니다.
					goodNumberCount++;
					break; // 하나의 "좋은 수"를 찾았으므로 더 이상 탐색할 필요 없이 다음 타겟으로 넘어갑니다.
				} else if(currentSum < targetValue) {
					// 합이 타겟보다 작으면, 합을 늘리기 위해 left 포인터를 증가시킵니다.
					left++;
				} else { // currentSum > targetValue
					// 합이 타겟보다 크면, 합을 줄이기 위해 right 포인터를 감소시킵니다.
					right--;
				}
			}
		}
		
		System.out.println(goodNumberCount);
	}	
}
```

### 5. 왜 이 방법이 최선인지 설명

주어진 문제(BOJ 1253)의 핵심은 **"배열 내의 어떤 숫자 `A`가 배열 내의 `B`와 `C` (단, `A`, `B`, `C`는 서로 다른 인덱스의 숫자여야 함)의 합으로 표현될 수 있는가?"**를 찾는 것입니다.

1.  **시간 복잡도 (O(N^2))**:
    *   **정렬 (O(N log N))**: 투 포인터 기법을 효율적으로 사용하기 위해서는 배열이 반드시 정렬되어야 합니다. `Arrays.sort()`는 N개의 요소를 정렬하는 데 O(N log N) 시간이 걸립니다.
    *   **외부 루프 (O(N))**: 배열의 모든 숫자 `numArr[i]`가 "좋은 수"가 될 수 있는 후보이므로, 각 숫자를 한 번씩 타겟으로 검사해야 합니다. 이는 O(N)번의 반복을 의미합니다.
    *   **내부 투 포인터 (O(N))**: 정렬된 배열에서 두 숫자의 합이 특정 값(`targetValue`)이 되는지 찾는 데 투 포인터 기법은 O(N)의 시간 복잡도를 가집니다. `left`와 `right` 포인터가 배열의 양 끝에서 시작하여 서로를 향해 이동하므로, 전체 배열을 한 번 순회하는 것과 같습니다.
    *   **결합**: 따라서 총 시간 복잡도는 O(N log N + N * N) = O(N^2)이 됩니다.
    *   **N=2000일 때**: N^2는 4,000,000입니다. 현대 컴퓨터에서 이 정도 연산은 일반적으로 수백 밀리초에서 1-2초 이내에 완료되므로, 문제의 시간 제한(보통 1~2초)을 충분히 만족합니다.
    *   **더 빠른 방법의 부재**: 이 문제의 특성상(세 개의 *다른* 인덱스에 해당하는 숫자를 찾는 것) O(N log N) 또는 O(N)으로 일반화된 솔루션을 찾기 어렵습니다. 해시맵을 사용할 경우 `O(N)`에 두 수의 합을 찾을 수 있지만, 세 인덱스가 모두 달라야 한다는 조건을 효율적으로 처리하기가 까다롭습니다. 정렬된 배열에서 투 포인터는 이 조건을 자연스럽게 처리하면서도 N을 곱했을 때 N^2라는 최적의 결과를 내는 방법입니다.

2.  **공간 복잡도 (O(N))**:
    *   입력 배열 `numArr`을 저장하는 데 O(N) 공간이 필요합니다.
    *   `Arrays.sort()`는 내부적으로 O(log N) 또는 O(N)의 추가 공간을 사용할 수 있지만, 이는 `numArr` 자체의 O(N)에 비하면 미미하거나, 원시 타입의 경우 대부분 in-place 정렬에 가까워 추가 공간이 크지 않습니다.
    *   따라서 O(N) 공간 복잡도는 입력 데이터를 저장하는 데 필수적이므로 최선입니다.

3.  **가독성**:
    *   코드에 명확한 변수명 (`goodNumberCount`, `targetValue`, `currentSum`)과 주석을 추가하여 각 부분의 역할을 설명했습니다.
    *   `Scanner` 자원 해제 (`sc.close()`)를 추가하여 리소스 관리를 개선했습니다.
    *   전반적인 구조가 직관적이어서 다른 사람이 코드를 이해하기 쉽습니다.

결론적으로, 주어진 제약 조건과 문제의 특성을 고려할 때, 정렬된 배열에서 투 포인터를 사용하는 O(N^2) 알고리즘은 **효율성, 정확성, 그리고 구현의 용이성** 측면에서 이 문제에 대한 최선의 접근 방식입니다.

# 코드 리뷰

**파일**: 이원준/202603/05/BOJ_17135.java

제공해주신 BOJ 17135번 문제 풀이에 대한 리뷰입니다. 전반적으로 깔끔하고 문제의 핵심 로직을 잘 구현하셨습니다. N, M이 작기 때문에 현재 방식으로도 충분히 통과할 수 있습니다.

---

### 1. 시간/공간 복잡도

*   **시간 복잡도:**
    *   `combination` 함수: M개의 열 중 3개의 궁수 위치를 선택하는 조합의 수만큼 `playGame` 함수가 호출됩니다. 이는 `M C 3` 입니다. `M C 3 = M * (M-1) * (M-2) / 6` 입니다. (최대 `15 C 3 = 455`)
    *   `playGame` 함수:
        *   원본 맵 복사: `O(N*M)`
        *   `N`번의 턴 동안 반복:
            *   `findTarget` 호출: 3명의 궁수 각각이 `O(N*M)`으로 전체 맵을 탐색합니다. 따라서 `O(3 * N*M)` = `O(N*M)`.
            *   타겟 처리: `targets` 리스트에는 최대 3개의 요소가 있으므로 `O(1)`.
            *   적 이동: 맵의 모든 요소를 한 칸씩 이동시킵니다. `O(N*M)`.
        *   따라서 한 번의 `playGame` 호출은 `N * (O(N*M) + O(N*M)) = O(N^2 * M)`의 시간 복잡도를 가집니다.
    *   **총 시간 복잡도:** `O(M^3 * N^2 * M)` = `O(M^4 * N^2)`.
        *   최대 값: `15^4 * 15^2 = 15^6 = 11,390,625`. (약 1.1 * 10^7)
        *   이는 1초 시간 제한 내에 충분히 통과할 수 있는 수준입니다.

*   **공간 복잡도:**
    *   `defence`, `gameStart`: 각각 `O(N*M)` 크기의 2차원 배열.
    *   `archers`, `targets`: `O(1)` (고정된 작은 크기).
    *   재귀 스택 (combination): 깊이가 3이므로 `O(1)`.
    *   **총 공간 복잡도:** `O(N*M)`.
        *   최대 값: `15*15 = 225`. 매우 작습니다.

**결론**: 시간/공간 복잡도 모두 주어진 제약 조건(N, M <= 15)에서 매우 효율적으로 동작합니다.

### 2. 논리 오류 여부

꼼꼼하게 살펴보았을 때, **논리 오류는 없어 보입니다.**

*   **조합 생성 (`combination`)**: `depth == 3` 조건과 `i + 1`을 넘겨주는 방식으로 중복 없는 3개의 궁수 위치 조합을 정확히 생성합니다.
*   **게임 플레이 (`playGame`)**:
    *   `gameStart` 배열을 복사하여 원본 `defence` 맵을 손상시키지 않는 것은 좋은 접근입니다.
    *   `turn`을 `N`번 반복하여 시뮬레이션 턴 수를 정확히 맞춥니다.
    *   `targets` 리스트에 먼저 모든 궁수의 타겟을 모으고, 그 후에 실제 제거를 진행하는 방식은 여러 궁수가 한 적을 동시에 타겟팅하더라도 그 적이 한 번만 제거되도록 보장합니다. `if(gameStart[r][c] == 1)` 조건으로 이미 죽은 적을 다시 세지 않습니다.
    *   적 이동 로직 (`for(int i = N-1; i >= 1; i--)` ... `gameStart[0][j] = 0;`)은 올바르게 동작합니다. 맨 아래 행부터 위로 한 칸씩 덮어씌우고, 마지막으로 0번 행을 비웁니다.
*   **타겟 찾기 (`findTarget`)**:
    *   문제의 우선순위(`가장 가까운 적 -> 가장 왼쪽에 있는 적`)를 정확히 구현했습니다.
    *   `minDist`, `minR`, `minC`를 초기화하고, `j`를 `N-1`부터 `0`까지 (아래에서 위로), `k`를 `0`부터 `M-1`까지 (왼쪽에서 오른쪽으로) 탐색합니다.
    *   `dist < minDist`는 가장 가까운 적을 찾고, `dist == minDist && k < minC`는 거리가 같을 때 가장 왼쪽에 있는 적을 찾습니다. 이 이중 조건문과 탐색 순서는 정확하게 작동합니다. 만약 거리가 같고 `k`도 같다면, `j`가 더 큰(더 아래에 있는) 적이 먼저 발견되므로 덮어씌워지지 않습니다. 이는 문제에서 요구하는 '가장 가까운 적 중 가장 왼쪽에 있는 적'에 부합합니다. (행은 동점처리 조건에 없으므로, 어떤 행이든 가장 가까우면 됩니다.)

### 3. 더 효율적인 방법

현재 풀이는 N, M이 작기 때문에 매우 효율적입니다. 하지만 N, M이 더 커졌을 경우를 가정한다면 몇 가지 고려해볼 수 있는 방법이 있습니다.

1.  **`findTarget` 최적화 (BFS/PriorityQueue)**:
    *   현재 `findTarget`은 각 궁수마다 전체 `N*M` 맵을 탐색하여 가장 가까운 적을 찾습니다. `D`의 범위가 작다는 것을 이용할 수 있습니다.
    *   궁수 위치 `(N, archerC)`에서 시작하여 BFS를 수행하면, `D` 범위 내의 적만 탐색할 수 있습니다. 이때, `PriorityQueue`에 `[거리, 열, 행]` 순서로 정렬되도록 넣어주면 가장 먼저 뽑히는 적이 원하는 타겟이 됩니다.
    *   하지만 N, M이 15로 작기 때문에, BFS의 큐/방문 배열 관리 오버헤드가 단순 이중 루프보다 더 클 수 있습니다. 현재의 `O(NM)` 스캔은 매우 효율적입니다.

2.  **적 이동 로직 최적화 (논리적인 이동)**:
    *   현재는 `N*M` 배열을 매 턴마다 복사하여 적을 이동시킵니다.
    *   대안으로, `gameStart` 맵을 직접 이동시키는 대신, 맵의 '기준점'을 이동시키는 방식으로 시뮬레이션할 수 있습니다. 예를 들어, `current_start_row` 변수를 두어 턴마다 증가시키고, 실제 맵의 `(r, c)` 위치를 `(r + current_start_row, c)`로 간주하는 식입니다.
    *   하지만 이 방식은 `findTarget`에서 적의 실제 좌표를 계산하는 로직이 복잡해질 수 있고, 죽은 적을 표시하기 위한 별도의 자료구조(`Set<Point>`)나 맵을 사용해야 할 수 있습니다. N, M이 작을 때는 현재의 물리적인 배열 이동이 훨씬 간단하고 성능 저하도 미미합니다.

**결론**: 현재 코드의 효율성은 주어진 N, M 범위에서 매우 좋습니다. 위 방법들은 N, M이 훨씬 커질 때 고려해볼 만한 것이며, 이 문제에서는 큰 이점이 없을 것입니다.

### 4. 놓친 엣지 케이스

제공된 코드에서 특별히 놓친 엣지 케이스는 없어 보입니다.

*   **맵에 적이 없는 경우**: `findTarget`이 비어있는 `targets` 리스트를 반환할 것이고, `currentKills`는 0이 되어 올바르게 처리됩니다.
*   **모든 적이 일찍 죽는 경우**: 남은 턴 동안은 빈 맵에서 `findTarget`이 계속 빈 리스트를 반환하며 `currentKills`는 더 이상 증가하지 않습니다. `maxKills`는 최종적으로 올바르게 업데이트됩니다.
*   **`D` 값이 아주 작거나 큰 경우**: 맨해튼 거리 계산은 `D` 값에 관계없이 정확하며, `dist <= D` 조건도 올바르게 작동합니다.
*   **궁수가 적을 전혀 타겟팅할 수 없는 경우**: `minDist`가 `Integer.MAX_VALUE`로 남아 `targets`에 추가되지 않으므로 올바르게 처리됩니다.
*   **`M < 3`인 경우**: `combination` 함수는 `depth == 3`을 만족시키지 못해 `playGame`을 호출하지 않을 것입니다. (단, 문제의 `M` 제약 조건은 3 이상일 것입니다.)

### 5. 코드 개선점 (비판적이되 건설적으로)

훌륭한 코드이지만, 몇 가지 사소한 개선점을 제안합니다.

1.  **사용하지 않는 import 제거**:
    *   `java.util.Queue`
    *   `java.util.LinkedList`
    *   이 두 클래스는 코드 내에서 사용되지 않습니다. 불필요한 import는 코드의 가독성을 약간 저해하고, 아주 미세하게 컴파일 시간을 늘릴 수 있습니다. 제거하는 것이 좋습니다.

2.  **상수(Constants) 활용**:

            }

                int archerR = N; // N도 상수로 정의하는 것을 고려해볼 수 있습니다 (예: GRID_ROWS)

    *   전반적으로 변수명과 함수명이 직관적이고 가독성이 좋습니다. 특별히 개선할 부분은 없어 보입니다.

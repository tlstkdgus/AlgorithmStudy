# 3/4 회고

## “사람의 머리로 푸는 방법”과 “코드로 치기 편한 방법” 간의 괴리

- 문제를 접근하는 논리 자체는 괜찮게 접근하는 것 같은데, 문제는 그 접근 방식을 코드로 구현하기가 너무 어렵다는 점이다.
- 그 이유는 “사람의 머리로 똑똑하게 푸는 방법”이 항상 “코드로 치기 편한 방법”과 일치하지 않기 때문이다.

## 괴리를 줄이려면?

### “생성”과 “검증”을 철저하게 분리하라

- **사람의 생각(복잡한 구현)**: “조건에 맞는 애들만 골라서, 안되는 애들은 쳐내고, 똑똑하게 줄세워야지!”
- **컴퓨터의 생각(쉬운 구현)**: “일단 무식하게 다 만들어 놓고(생성), 조건에 안 맞는 걸 나중에 쳐내자(검증)”
- **솔루션**: 일단 “경우의 수를 찍어내는 공장(템플릿)”을 먼저 돌린다. 그리고 그게 완성된 시점(depth == 끝)에서 “검사관(BFS나 시뮬레이션)”을 호출해 조건에 맞는지 확인하는 투트랙으로 가야 코드가 꼬이지 않는다.

### 내가 아는 ‘템플릿’에 문제를 우겨넣어라

- **사람의 생각**: 문제 상황을 있는 그대로 코드로 번역하려고 함. (야구에서 1~3번 타자 뽑고, 남은 5명 줄 세우는 두 단계 재귀)
- **컴퓨터의 생각**: “이거 결국 ‘순서’가 필요하네? 무조건 기본 순열 템플릿!” / “이거 결국 O/X ‘팀 나누기’네? 무조건 부분집합 템플릿!”
- **솔루션**: 문제를 읽고 논리가 섰다면, 그걸 새로운 논리로 코딩하려 하지 말고, “이 문제는 내가 외우고 있는 순열, 조합, 부분집합, BFS 중 어떤 템플릿 스킨을 씌울까?”라는 마인드로 접근해야 해. 4번 타자 고정처럼 특수한 조건은 템플릿 안에 if 문 한 줄 얹는 식으로 해결하는 것이 베스트

### 하드코딩(if-else)를 ‘자료구조와 규칙’으로 대체하라

- **사람의 생각**: “1루타일 때, 2루타일 때, 3루타일 때 주자 이동을 다 다르게 적어줘야지.”
- **컴퓨터의 생각**: “배열을 3루부터 1루까지 역순으로 돌리면서 인덱스만 더해주면 한 번에 끝나는데?”
- **솔루션**: 조건이 복잡해질 것 같으면 손가락을 멈추고 1분만 생각해보자. 배열 인덱스, 나머지 연산자(%), 큐(Queue), 덱(Deque) 같은 자료구조나 수학적 규칙을 쓰면 수십 줄의 if-else가 한 줄짜리 for문으로 마법처럼 압축되는 경우가 많다.

## 3분 설계 체크리스트

- **메인 무기 선택**: 이건 순열인가? 조합인가? 부분집합인가? BFS인가?
- **데이터의 형태 (상태 관리)**: 지도(2차원 배열)인가? 1차원 방문 배열(boolean[])인가?
- **가장 골치 아픈 조건**: (ex. 4번 타자 고정) 이걸 무기(템플릿)의 어디에 한 줄로 끼워 넣을 것인가?

# 최적화 분석

주어진 코드는 백트래킹(순열)과 시뮬레이션을 결합한 문제입니다. 핵심은 9명의 타자 중 1번 타자가 4번 타순에 고정되어 있다는 조건 하에 나머지 8명의 타자 순서를 정하고, 각 순서에 대해 게임을 시뮬레이션하여 최대 점수를 찾는 것입니다.

현재 코드의 시간 복잡도와 공간 복잡도는 문제의 제약 조건(N <= 50)을 고려할 때 이미 효율적인 편입니다. `N`이 50일 때 8! * N * (타자 수) 정도의 연산이 이루어지는데, 이는 약 4만 * 50 * 9 = 1,800만 정도의 연산으로 충분히 허용 범위 내입니다. 따라서 시간 복잡도를 획기적으로 줄이는 것은 어렵습니다.

주요 개선점은 `playGame` 메서드 내의 복잡한 조건문들을 좀 더 일반화하고, 이를 통해 가독성을 높이며 잠재적인 실수 가능성을 줄이는 것입니다.

---

### 1. 시간 복잡도 개선 방법

현재 알고리즘의 시간 복잡도는 `makeLineup` 메서드에서 8! (나머지 8명의 타자 순열)을 생성하고, 각 순열에 대해 `playGame` 메서드에서 N이닝 동안 게임을 시뮬레이션하는 구조입니다.
*   **순열 생성**: 8! = 40,320가지
*   **게임 시뮬레이션**: N 이닝 * (이닝당 최대 타자 수). 이닝당 최대 타자 수는 3아웃이 나기 전까지 무한히 나올 수 있지만, 현실적으로는 한 이닝에 9명 모두 타석에 들어서기 전에 3아웃이 나는 경우가 대부분이며, 최악의 경우 한 이닝에 점수가 계속 나도 3아웃이 나기 전까지 대략 9명 안팎으로 타자가 들어섭니다. (3아웃 * 3루 가득 채움 = 9명)
따라서 총 시간 복잡도는 약 **O(8! * N * (타자 수))** 입니다.

**개선 방안**:
이 문제의 본질이 모든 가능한 타순을 시뮬레이션하는 것이기 때문에, 8! 가지의 순열을 생성하는 것은 피할 수 없습니다. N 이닝을 시뮬레이션하는 것 역시 필수적입니다. 따라서, **본질적인 시간 복잡도를 개선할 방법은 없습니다.** 현재의 `N` 값 (최대 50)에서는 1.8 * 10^7 정도의 연산으로 충분히 빠르므로, 시간 초과 걱정은 없습니다.

### 2. 공간 복잡도 개선 방법

*   `lineup`: 10개 (int)
*   `visited`: 10개 (boolean)
*   `result`: (N+1) * 10개 (int)
*   `base`: 4개 (boolean)

**개선 방안**:
사용하는 배열들의 크기가 매우 작고, N의 크기에 비례하는 `result` 배열도 최대 `51 * 10` 크기로 매우 작습니다. 따라서 **공간 복잡도 측면에서도 특별히 개선할 부분은 없습니다.** 현재 코드는 이미 효율적인 공간 활용을 하고 있습니다.

### 3. 가독성 개선

가장 크게 개선할 수 있는 부분은 `playGame` 메서드 내의 타구(안타, 2루타, 3루타, 홈런) 처리 로직입니다. 현재는 `if-else if` 문이 길게 늘어서 있어 다소 직관적이지 않고 반복적입니다. 이 부분을 일반화된 로직으로 변경하여 가독성을 높일 수 있습니다.

**개선 방안**:
주자 진루 로직을 다음과 같이 일반화할 수 있습니다.
1.  **기존 주자들의 진루 및 득점 처리**: 3루 주자부터 1루 주자까지 역순으로 `hit`만큼 진루시키고, 홈을 밟으면 득점 처리합니다.
2.  **타자의 진루 및 득점 처리**: 타자 본인이 `hit`만큼 진루합니다. 홈런일 경우 즉시 득점하고, 아닐 경우 해당 루에 안착합니다.

### 4. 최종 최적화된 코드

```java
package simulation;

import java.util.Scanner;

public class BOJ_17281_Optimized {
    // 경기 결과를 저장하는 배열. result[이닝][선수번호] = 결과 (0: 아웃, 1: 안타, 2: 2루타, 3: 3루타, 4: 홈런)
    static int[][] result;
    // 타자 순서를 저장하는 배열. lineup[타순] = 선수번호
    static int[] lineup = new int[10];
    // 선수가 타순에 배정되었는지 확인하는 배열. visited[선수번호] = 배정 여부
    static boolean[] visited = new boolean[10];
    // 총 이닝 수
    static int N;
    // 최대 점수
    static int maxScore;

    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        N = sc.nextInt();

        result = new int[N + 1][10]; // 이닝은 1부터 N, 선수 번호는 1부터 9
        
        // 경기 결과 입력
        for (int i = 1; i <= N; i++) {
            for (int j = 1; j <= 9; j++) {
                result[i][j] = sc.nextInt();
            }
        }
        sc.close();

        // 1번 선수는 4번 타순에 고정
        lineup[4] = 1;
        visited[1] = true; // 1번 선수는 이미 타순에 배정됨

        maxScore = Integer.MIN_VALUE; // 최대 점수 초기화

        // 1번 타순부터 타자 배정 시작 (4번 타순은 이미 배정되었으므로 건너뜀)
        makeLineup(1);
        
        System.out.println(maxScore);
    }

    /**
     * 타순을 생성하는 백트래킹 메서드
     * @param depth 현재 배정할 타순 (1부터 9)
     */
    static void makeLineup(int depth) {
        // 모든 타순이 배정되었으면 게임을 시작하고 최대 점수 갱신
        if (depth == 10) {
            playGame();
            return;
        }

        // 4번 타순은 1번 선수가 고정되어 있으므로 건너뛰고 다음 타순으로 넘어감
        if (depth == 4) {
            makeLineup(depth + 1);
            return;
        }

        // 2번부터 9번 선수까지 순서대로 타순에 배정 시도
        for (int playerNum = 2; playerNum <= 9; playerNum++) {
            // 아직 타순에 배정되지 않은 선수라면
            if (!visited[playerNum]) {
                visited[playerNum] = true;        // 방문 처리
                lineup[depth] = playerNum;        // 현재 타순에 선수 배정
                
                makeLineup(depth + 1);            // 다음 타순 배정 재귀 호출
                
                visited[playerNum] = false;       // 백트래킹: 선수 방문 상태 해제
            }
        }
    }

    /**
     * 현재 생성된 타순으로 N 이닝 게임을 시뮬레이션하고 점수를 계산하는 메서드
     */
    static void playGame() {
        int currentScore = 0; // 현재 타순의 총 점수
        int currentBatterIdx = 1; // 현재 타석에 들어설 타자의 타순 (1부터 9)

        // 각 이닝 시뮬레이션
        for (int inning = 1; inning <= N; inning++) {
            int outs = 0; // 현재 이닝의 아웃 카운트
            // 베이스 상태. base[1]: 1루, base[2]: 2루, base[3]: 3루 (true: 주자 있음, false: 주자 없음)
            boolean[] bases = new boolean[4]; 

            // 3아웃이 되기 전까지 이닝 진행
            while (outs < 3) {
                int currentPlayer = lineup[currentBatterIdx]; // 현재 타석의 선수 번호
                int hitResult = result[inning][currentPlayer]; // 해당 이닝 해당 선수의 타격 결과

                if (hitResult == 0) { // 아웃
                    outs++;
                } else { // 안타, 2루타, 3루타, 홈런
                    // 1. 기존 주자들의 진루 및 득점 처리
                    // 3루 주자부터 1루 주자까지 역순으로 처리해야 올바른 진루가 보장됨
                    for (int b = 3; b >= 1; b--) {
                        if (bases[b]) { // 해당 베이스에 주자가 있다면
                            bases[b] = false; // 현재 베이스에서 주자 제거
                            if (b + hitResult >= 4) { // 주자가 홈을 밟으면 득점
                                currentScore++;
                            } else { // 홈을 밟지 않으면 이동한 베이스에 안착
                                bases[b + hitResult] = true;
                            }
                        }
                    }

                    // 2. 타자의 진루 및 득점 처리
                    if (hitResult == 4) { // 홈런이면 타자 본인도 득점
                        currentScore++;
                    } else { // 안타, 2루타, 3루타면 해당 베이스에 타자 안착
                        bases[hitResult] = true;
                    }
                }

                // 다음 타자 타순으로 이동
                currentBatterIdx++;
                if (currentBatterIdx > 9) {
                    currentBatterIdx = 1; // 9번 타자 다음은 1번 타자
                }
            }
        }
        // 모든 이닝 종료 후, 최대 점수 갱신
        maxScore = Math.max(maxScore, currentScore);
    }
}
```

### 5. 왜 이 방법이 최선인지 설명

1.  **문제의 본질적 제약**:
    *   **순열 생성**: 9명의 타자 중 1명이 고정된 상태에서 나머지 8명의 타순을 정하는 것은 8! 가지의 모든 경우의 수를 탐색해야 합니다. 이는 문제에서 요구하는 '최대 점수'를 찾기 위한 필수적인 과정이므로, 이 부분의 시간 복잡도(O(8!))는 줄일 수 없습니다.
    *   **시뮬레이션**: 각 타순에 대해 N 이닝을 시뮬레이션하는 과정 역시 필수적입니다. 이 부분의 시간 복잡도(O(N * (이닝당 타자 수)))도 줄일 수 없습니다.

2.  **현재 N 값에 대한 효율성**:
    *   `N`이 최대 50일 때, 총 연산 수는 약 `8! * N * 9` = `40320 * 50 * 9` = 약 1.8 * 10^7 (1천 8백만)입니다. 이는 일반적으로 1~2초의 시간 제한을 가진 온라인 저지 환경에서 충분히 허용되는 연산량입니다. 따라서, 더 복잡한 자료구조나 알고리즘을 도입하여 시간 복잡도를 미세하게 줄이려 하기보다, 현재의 백트래킹 + 시뮬레이션 방식이 가장 적절합니다.

3.  **가독성 및 유지보수성 향상**:
    *   원래 코드의 `playGame` 메서드에서 안타 종류(1루타, 2루타, 3루타, 홈런)에 따라 베이스 주자들을 수동으로 이동시키고 득점을 계산하는 `if-else if` 문은 길고 반복적이며 오류를 유발할 가능성이 있었습니다.
    *   최적화된 코드에서는 이 부분을 **일반화된 진루 로직**으로 변경했습니다.
        *   `for (int b = 3; b >= 1; b--)` 루프를 통해 3루 주자부터 역순으로 처리함으로써, 앞선 주자의 이동이 뒤선 주자의 베이스에 영향을 주지 않도록 합니다 (예: 1루 주자가 2루로 이동한 후, 새로운 타자가 1루에 안착).
        *   `b + hitResult >= 4` 조건으로 주자가 홈을 밟았는지 판별하고 득점합니다.
        *   타자 본인의 진루(`if (hitResult == 4)` 또는 `bases[hitResult] = true;`)는 기존 주자 처리 후에 수행됩니다.
    *   이러한 일반화된 로직은 코드의 길이를 줄이고, 어떤 종류의 안타가 발생하더라도 동일한 방식으로 처리되므로 **가독성이 뛰어나고 유지보수가 용이**하며 **논리적 오류 발생 가능성을 낮춥니다.**

결론적으로, 이 코드는 문제의 제약 조건과 요구 사항을 만족하면서, 시간 및 공간 효율성 측면에서는 이미 최적에 가깝고, 가독성 측면에서는 주자 진루 로직을 일반화함으로써 크게 개선되었습니다. 이는 주어진 문제에 대한 가장 합리적이고 효율적인 해결책입니다.

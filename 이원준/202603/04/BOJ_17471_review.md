# 코드 리뷰

**파일**: 이원준/202603/04/BOJ_17471.java

제공해주신 알고리즘 풀이는 백준 17471번 "게리맨더링" 문제에 대한 표준적이고 잘 구현된 풀이입니다. 문제의 제약사항(N이 최대 10)을 고려할 때, 현재의 접근 방식은 매우 적절합니다. 비판적이고 건설적인 관점에서 분석해 보겠습니다.

---

### 1. 시간/공간 복잡도

*   **시간 복잡도:** `O(2^N * (N + E))`
    *   `makeSubset`: `N`개의 구역을 두 팀으로 나누는 모든 경우의 수를 탐색합니다. 이는 재귀 호출 트리의 깊이가 `N`이고 각 노드마다 두 갈래로 나뉘므로 총 `2^N`가지의 조합을 생성합니다.
    *   `isConnected`: 각 조합에 대해 두 번 호출됩니다. `isConnected` 함수는 BFS(너비 우선 탐색)를 사용하여 연결성을 확인합니다. BFS의 시간 복잡도는 인접 리스트를 사용할 경우 `O(V + E)`입니다. 여기서는 `V`가 `N` (도시의 개수)이고 `E`는 연결된 간선의 개수입니다. `targetCount`를 세는 루프도 `O(N)`이므로, `isConnected`는 `O(N + E)`입니다.
    *   `calculatePopulationDifference`: 각 조합에 대해 한 번 호출됩니다. `N`개의 구역 인구를 합산하므로 `O(N)`입니다.
    *   종합적으로, `2^N`번의 조합마다 `O(N + E)` 연산을 수행하므로, 총 시간 복잡도는 `O(2^N * (N + E))`가 됩니다.
    *   문제에서 `N`이 최대 10이므로 `2^10 = 1024`입니다. `E`는 최대 `N*(N-1)/2` (N=10일 때 45)이므로 `N+E`는 최대 `10+45 = 55` 정도입니다. `1024 * 55`는 약 56,000으로, 현대 컴퓨터에서 매우 빠르게 수행될 수 있는 연산량입니다.

*   **공간 복잡도:** `O(N + E)`
    *   `population`: `O(N)`
    *   `adjList`: 인접 리스트는 모든 간선을 저장하므로 `O(N + E)`입니다.
    *   `isSelected`: `O(N)`
    *   `makeSubset` 재귀 스택: 재귀의 최대 깊이는 `N`이므로 `O(N)`입니다.
    *   `isConnected` 내부 `queue` 및 `visited` 배열: `O(N)`
    *   종합적으로, 가장 큰 부분을 차지하는 인접 리스트에 의해 `O(N + E)`가 됩니다. 이 또한 매우 효율적인 공간 사용입니다.

### 2. 논리 오류

제출하신 코드에는 **명백한 논리 오류는 없습니다.**
핵심 로직들이 문제의 요구사항을 잘 반영하고 있습니다.

*   **모든 부분집합 생성:** `makeSubset` 함수는 모든 가능한 두 구역 분할 조합(공집합 포함)을 정확히 생성합니다.
*   **연결성 확인:** `isConnected` 함수는 BFS를 사용하여 특정 팀에 속한 도시들이 모두 서로 연결되어 있는지 (하나의 컴포넌트를 이루는지) 정확하게 확인합니다.
    *   `targetCount == 0`인 경우 `false`를 반환하여, 비어있는 팀은 유효하지 않음을 처리합니다. 이는 필수적인 조건입니다.
*   **인구 차이 계산 및 최소값 갱신:** `calculatePopulationDifference`는 두 팀의 인구 합을 계산하고, 현재의 최소 차이 `minDiff`를 올바르게 갱신합니다.
*   **불가능한 경우 처리:** 초기 `minDiff`를 `Integer.MAX_VALUE`로 설정하고, 최종적으로 이 값이 변하지 않았다면 `-1`을 출력하는 방식으로 유효한 분할이 없는 경우를 잘 처리합니다. (예: N=1인 경우, 어떤 방식으로 나누든 한 쪽이 비어있게 되므로 항상 -1이 나와야 합니다.)

### 3. 더 효율적인 방법

`N`의 제약 조건(최대 10)을 고려할 때, 현재의 `O(2^N * (N + E))` 브루트 포스(Brute Force) 방식이 가장 일반적이고 효율적인 접근 방식입니다. `N`이 작은 경우, 이 방식은 종종 최적의 해를 찾는 가장 간단하고 빠른 방법입니다.

만약 `N`이 훨씬 크다면 (예: 20 이상), 다른 접근 방식을 고려해야 할 것입니다:

*   **Meet-in-the-Middle:** `N`이 약 40 정도일 때 `2^(N/2)` 방식이 사용될 수 있지만, 이 문제의 "두 팀이 모두 연결되어야 한다"는 제약 때문에 단순히 두 팀의 인구 합을 맞추는 것보다 복잡해집니다. 각 부분집합의 연결성 정보를 어떻게 효율적으로 저장하고 결합할지가 관건이 됩니다.
*   **네트워크 플로우 / 최소 컷:** 그래프를 두 부분으로 나누는 문제에서 간혹 등장하지만, 이 문제의 핵심은 "두 부분이 모두 연결되어야 한다"는 점이며, 최소 컷은 단순히 연결을 끊는 최소 비용을 찾을 뿐이지 끊어진 두 부분이 각각 연결성을 유지하는지는 보장하지 않습니다. 이 문제에 직접 적용하기는 어렵습니다.
*   **동적 프로그래밍:** 현재 문제 구조에서는 부분 문제 정의가 어려워 보입니다. "어떤 상태를 어떻게 저장해야 다음 상태를 효율적으로 계산할 수 있을까"에 대한 명확한 해답이 없습니다.

**결론적으로, 현재 문제의 `N` 범위에서는 지금의 완전 탐색(부분집합 생성 후 검증) 방식이 최적이며, 더 효율적인 일반적인 알고리즘은 없습니다.**

### 4. 놓친 엣지 케이스

제공해주신 코드는 대부분의 엣지 케이스를 잘 처리하고 있습니다.

*   **`N=1`인 경우:**
    *   `makeSubset`이 `depth == N+1`에 도달하면 `isSelected[1]`이 `true` 또는 `false`인 두 가지 상태를 확인합니다.
    *   `isSelected[1] == true`: A팀에 도시 1, B팀은 비어있음. `isBConnected(false)`에서 `targetCount == 0`이므로 `false` 반환. 조건 불만족.
    *   `isSelected[1] == false`: B팀에 도시 1, A팀은 비어있음. `isAConnected(true)`에서 `targetCount == 0`이므로 `false` 반환. 조건 불만족.
    *   결과적으로 `minDiff`는 `Integer.MAX_VALUE`로 남아있고, `-1`이 출력됩니다. **이는 정확한 처리입니다.** N=1인 경우 두 개의 비어있지 않고 연결된 구역으로 나눌 수 없습니다.

*   **그래프가 처음에 연결되어 있지 않은 경우:**
    *   예를 들어, 도시 1-2-3이 연결되어 있고 도시 4-5가 연결되어 있지만 두 그룹 간에 간선이 없는 경우.
    *   만약 {1,2,3}이 A팀, {4,5}가 B팀으로 나뉜다면, `isAConnected(true)` (BFS on {1,2,3})는 `true`를 반환할 것이고, `isBConnected(false)` (BFS on {4,5})도 `true`를 반환할 것입니다. 이 경우 유효한 분할로 처리됩니다.
    *   만약 {1,2}가 A팀, {3,4,5}가 B팀으로 나뉜다면, `isBConnected(false)`는 도시 3,4,5가 하나의 연결된 컴포넌트를 이루지 못하므로 `false`를 반환할 것입니다.
    *   이러한 경우들을 `isConnected` 함수가 잘 처리합니다. **문제 없습니다.**

*   **모든 도시가 서로 고립된 경우 (간선이 없는 경우):**
    *   `adjList`의 모든 리스트가 비어있는 경우.
    *   `N >= 2`라면, `isAConnected` 함수는 어떤 팀이든 `targetCount > 1`인 경우 `startNode`에서 시작해서 다른 노드로 갈 수 없으므로 `count`가 `targetCount`보다 작게 됩니다. 따라서 `false`를 반환할 것입니다.
    *   유일하게 `true`가 될 수 있는 경우는 `targetCount == 1`일 때입니다 (단일 도시로 이루어진 팀).
    *   만약 N=2, 도시 1,2 인구가 P1, P2이고 간선이 없다면:
        *   {1}을 A팀, {2}를 B팀으로 나눌 때, `isAConnected(true)` (for {1})는 `true`, `isBConnected(false)` (for {2})도 `true`를 반환합니다. 이 경우 `|P1-P2|`가 `minDiff`로 갱신됩니다.
    *   이 또한 **정확한 처리입니다.**

### 5. 코드 개선점

대부분의 코드가 깔끔하고 가독성이 좋습니다. 몇 가지 사소한 개선점은 다음과 같습니다.

1.  **제네릭 배열 경고 회피 (옵션):**
    ```java
    // 현재 코드: 경고 발생 가능 (raw type Array creation)
    // adjList = new ArrayList[N + 1];

    // 개선: 명시적으로 캐스팅하거나 List<List<Integer>> 사용 (선택 사항)
    adjList = (ArrayList<Integer>[]) new ArrayList[N + 1];
    // 또는
    // List<List<Integer>> adjList = new ArrayList<>();
    // for(int i=0; i<=N; i++) adjList.add(new ArrayList<>());
    ```
    이 부분은 경쟁 프로그래밍에서는 흔히 무시되는 경고이지만, 정식 프로젝트 코드에서는 처리하는 것이 좋습니다. 현재 방식도 기능적으로는 전혀 문제가 없습니다.

2.  **클래스 변수 초기화 위치 (선택 사항):**
    `minDiff`는 `Integer.MAX_VALUE`로 잘 초기화되었습니다. `isSelected` 배열은 `main` 함수에서 `new boolean[N+1]`로 초기화되는데, 이 때 기본값인 `false`로 채워집니다. 이는 `isSelected[0]`을 사용하지 않는 것을 고려하면 논리적으로도 깔끔합니다. 별다른 변경 필요는 없습니다.

3.  **가독성 향상을 위한 상수 사용 (선택 사항):**
    `isSelected[i] == targetTeam`과 같이 `boolean` 값을 직접 비교하는 것은 명확하지만, `true`를 "팀 A", `false`를 "팀 B"로 생각한다면 다음과 같이 명시적인 상수를 사용하면 조금 더 읽기 쉬울 수 있습니다.
    ```java
    // class variables
    static final boolean TEAM_A = true;
    static final boolean TEAM_B = false;

    // makeSubset 내부
    isSelected[depth] = TEAM_A;
    makeSubset(depth+1);
    isSelected[depth] = TEAM_B;
    makeSubset(depth+1);

    // isConnected 내부
    // isConnected(TEAM_A) 또는 isConnected(TEAM_B)

    // calculatePopulationDifference 내부
    if(isSelected[i] == TEAM_A) { // 또는 if(isSelected[i])
        Asum += population[i];
    } else { // isSelected[i] == TEAM_B
        Bsum += population[i];
    }
    ```
    이는 순전히 가독성 개선 목적이며, 현재 코드도 충분히 이해하기 쉽습니다.

4.  **불필요한 `else` 제거 (사소):**
    `calculatePopulationDifference` 함수에서 `if(isSelected[i])` 이후의 `else` 블록은 `isSelected[i]`가 `false`일 때만 실행되므로, `else if (!isSelected[i])` 와 같은 추가 조건 검사는 불필요합니다. 현재 코드가 이미 그렇게 되어있어 문제가 없습니다.

# 코드 리뷰

**파일**: 이원준/202603/04/BOJ_17281.java

제공해주신 알고리즘 풀이(BOJ 17281, 야구)에 대해 비판적이면서도 건설적인 리뷰를 시작하겠습니다. 전반적으로 잘 작성된 코드이며, 문제의 핵심을 정확히 짚어 구현하신 것으로 보입니다.

---

### 1. 시간/공간 복잡도

**시간 복잡도:**
*   **타순 생성 (makeLineup):** 9명의 선수 중 1번 선수가 4번 타순에 고정되므로, 나머지 8명의 선수를 8개의 남은 타순에 배치하는 경우의 수는 `8!` (8 팩토리얼) 입니다. `8! = 40,320`으로 매우 작은 숫자입니다.
*   **게임 시뮬레이션 (playGame):**
    *   `N` 이닝 동안 게임을 진행합니다.
    *   각 이닝에서는 최대 3아웃이 될 때까지 타자들이 타석에 들어섭니다. 한 이닝에 최대 9명의 타자가 모두 아웃될 수 있으므로, 최악의 경우 한 이닝당 대략 9번의 타격이 발생할 수 있습니다. (실제로는 3아웃이 되려면 최소 3번의 타격이 필요하며, 최대는 3*(9-1)+3 = 27번 정도까지 가능하지만, 여기서는 9명 모두 한 번씩 돈다고 가정해도 무방합니다.)
    *   타격 한 번당 베이스 진루 및 득점 계산은 `O(1)` 상수로 처리됩니다.
    *   따라서 `playGame` 함수의 시간 복잡도는 `O(N * (이닝당 타자 수))`이며, 이는 대략 `O(N * 9)` 또는 간단히 `O(N)`으로 볼 수 있습니다.
*   **총 시간 복잡도:** `O(8! * N)`
    *   `N`은 최대 50이므로, `40,320 * 50 = 2,016,000` 입니다. 이는 1초 이내에 충분히 실행 가능한 연산량입니다 (일반적으로 1억 연산에 1초).

**공간 복잡도:**
*   `lineup`, `visited`: `O(10)` (상수 크기) -> `O(1)`
*   `N`, `maxScore`: `O(1)`
*   `result`: `O(N * 10)` -> `O(N)` (입력 크기에 비례)
*   재귀 스택 (makeLineup): 최대 깊이 9 -> `O(1)`
*   `base`: `O(4)` (상수 크기) -> `O(1)`
*   **총 공간 복잡도:** `O(N)`
    *   `N`은 최대 50이므로, `O(50 * 10)` 크기의 2차원 배열은 매우 작은 공간을 차지합니다.

**결론:** 시간/공간 복잡도 모두 주어진 제약 조건 내에서 매우 효율적으로 동작하며, 문제 해결에 충분합니다.

### 2. 논리 오류 여부

꼼꼼히 검토한 결과, **주요한 논리 오류는 발견되지 않았습니다.**

*   **타순 생성 (`makeLineup`):**
    *   1번 선수를 4번 타순에 고정하는 로직 (`lineup[4] = 1; visited[1] = true;` 후 `depth == 4`일 때 `makeLineup(depth + 1); return;`)이 정확합니다.
    *   나머지 8명의 선수를 배치하는 순열 로직도 올바르게 구현되었습니다.
*   **게임 시뮬레이션 (`playGame`):**
    *   아웃 카운트, 베이스 초기화, 이닝 종료 조건 (3아웃) 등이 올바르게 처리됩니다.
    *   `batterIdx`가 9를 넘어가면 1로 초기화되는 순환 방식도 정확합니다.
    *   각 타격 결과 (안타, 2루타, 3루타, 홈런)에 따른 주자 진루 및 득점 계산 로직도 정확합니다. 3루 주자가 홈으로 들어오고, 2루 주자가 3루로, 1루 주자가 2루로 가는 등, 각 베이스의 주자가 정확히 이동합니다.
    *   `maxScore` 업데이트도 올바르게 진행됩니다.

### 3. 더 효율적인 방법

현재 구현된 방식(`8!` 순열 + `N` 이닝 시뮬레이션)은 이 문제의 표준적이고 가장 효율적인 접근 방식입니다. `N`과 `9`라는 작은 숫자를 고려할 때, 이보다 더 근본적으로 효율적인 알고리즘은 찾기 어렵습니다.

다만, `playGame` 내에서 베이스 진루 로직을 조금 더 간결하게 만들 수는 있습니다. 현재 코드는 `hit == 1`, `hit == 2`, `hit == 3`, `hit == 4` 각각에 대해 주자 이동 및 득점 로직을 거의 복사/붙여넣기 한 형태입니다. 이는 가독성 측면에서 미묘하게 반복적이고, 만약 베이스 이동 규칙이 복잡해지면 버그를 유발하기 쉬울 수 있습니다.

**개선 아이디어 (로직 간결화):**
주자 이동 로직을 아래와 같이 구조화하면 중복을 줄일 수 있습니다.
1.  현재 베이스에 있는 주자들이 이동하여 득점하거나 다음 베이스로 진루합니다. (3루부터 1루까지 역순으로 처리해야 충돌 없이 정확히 처리 가능)
2.  타자가 해당 베이스로 진루하거나 득점합니다.

```java
// playGame 함수 내부, hit > 0 인 경우
else {
    // 1. 기존 주자들의 이동 및 득점 처리 (3루 -> 2루 -> 1루 순으로 처리해야 겹치지 않음)
    // 3루 주자 처리
    if (base[3]) {
        score++; // 3루 주자는 무조건 득점 (1,2,3,4루타 모두)
        base[3] = false;
    }
    // 2루 주자 처리
    if (base[2]) {
        if (hit >= 2) { // 2루타 이상이면 득점
            score++;
        } else { // 1루타면 3루로 진루
            base[3] = true;
        }
        base[2] = false;
    }
    // 1루 주자 처리
    if (base[1]) {
        if (hit >= 3) { // 3루타 이상이면 득점
            score++;
        } else { // 1루타면 2루로, 2루타면 3루로 진루
            base[1 + hit] = true;
        }
        base[1] = false;
    }

    // 2. 타자의 진루 및 득점 처리
    if (hit == 4) { // 홈런이면 타자도 득점
        score++;
    } else { // 홈런이 아니면 해당 베이스로 진루
        base[hit] = true;
    }
}
```
**주의:** 위의 개선 아이디어는 실제 야구 규칙과 주자 이동 로직을 더 면밀히 따져봐야 합니다. 예를 들어, 1루타 상황에서 1루 주자가 2루로, 2루 주자가 3루로, 3루 주자가 홈으로 가는 경우, `base[i+hit]`으로 단순히 더해서 처리하면 문제가 생길 수 있습니다 (즉, `base[1]`이 `true`일 때 `base[2]`도 `true`로 만들 수 있는데, 그 전에 `base[2]`가 `true`였던 주자가 이미 3루로 가야 합니다). 현재 코드가 이 부분을 명시적으로 처리하여 오히려 더 정확할 수 있습니다.

**결론적으로, 현재 코드는 정확성을 보장하기 위해 각 케이스를 명확히 분리한 형태이며, 이는 나쁘지 않은 선택입니다. 위와 같은 '간결화' 시도는 오히려 버그를 유발할 수 있으므로, 현재 코드의 명확한 `if-else if` 구조를 유지하는 것이 더 좋습니다.**

### 4. 놓친 엣지 케이스

현재 코드는 대부분의 엣지 케이스를 잘 처리하고 있습니다.
*   `N = 1` (최소 이닝 수): `for(int inning = 1; inning <= N; inning++)` 루프가 정확히 한 번 실행되어 문제가 없습니다.
*   모든 타자가 홈런만 치는 경우: 득점 계산이 정확합니다.
*   모든 타자가 아웃만 당하는 경우: 점수가 0으로 유지되며, `maxScore`는 `Integer.MIN_VALUE` 또는 초기화된 0 (만약 `maxScore`가 0으로 초기화되었다면)이 됩니다.
*   타순이 한 바퀴를 돌지 않고 이닝이 끝나는 경우: `batterIdx`가 잘 유지되어 다음 이닝으로 넘어갑니다.
*   `maxScore` 초기화가 `Integer.MIN_VALUE`로 잘 되어 있어, 모든 시뮬레이션 결과가 0이라도 정확한 최댓값을 찾을 수 있습니다.

놓친 엣지 케이스는 특별히 보이지 않습니다.

### 5. 코드 개선점

대부분의 로직은 훌륭하게 구현되었지만, 몇 가지 사소한 개선점을 제안합니다.

1.  **상수 활용:**
    *   마법의 숫자(magic number)를 상수로 정의하여 가독성을 높일 수 있습니다. 예를 들어, 타자 수, 아웃 수, 1번 선수의 타순 등.
    ```java
    // class BOJ_17281 내부에
    final static int NUM_PLAYERS = 9;
    final static int PLAYER_ONE_ID = 1;
    final static int PLAYER_ONE_BATTING_ORDER = 4; // 1번 선수는 4번 타자
    final static int MAX_OUTS = 3;
    final static int MAX_BASES = 3; // 1루, 2루, 3루
    ```
    이후 코드에서 `9`, `3`, `4` 등의 숫자를 이 상수로 대체하면 됩니다. (예: `for(int depth = 1; depth <= NUM_PLAYERS; depth++)` 또는 `while(outs < MAX_OUTS)`)

2.  **`visited` 배열 크기 및 초기화:**
    *   `visited` 배열은 `new boolean[10]`으로 선언되었지만, 실제 사용되는 인덱스는 1부터 9까지입니다. 명시적으로 `new boolean[NUM_PLAYERS + 1]`과 같이 선언하면 의도를 더 명확히 할 수 있습니다.
    *   `main` 함수에서 `visited[1] = true;`를 하는 것은 1번 선수를 이미 배치했음을 표시하는 좋은 방법입니다.

3.  **`makeLineup`의 재귀 조건 정리:**
    *   현재 `makeLineup`은 `depth == 10`이 종료 조건이지만, `lineup` 배열의 인덱스는 1부터 9까지 사용합니다. `depth`가 `NUM_PLAYERS + 1` (즉, 10)일 때 모든 타순이 완성된 것으로 보는 것은 논리적입니다.
    *   `depth == 4` 조건은 1번 선수가 4번 타자로 고정된 것을 처리하는데, 이 부분은 문제의 핵심 제약 사항이므로 명확하게 유지하는 것이 좋습니다.

4.  **`base` 배열 인덱싱 개선 (선택 사항):**
    *   현재 `boolean[] base = new boolean[4];`에서 `base[0]`은 사용하지 않고, `base[1]`이 1루, `base[2]`가 2루, `base[3]`이 3루를 나타냅니다. 이는 직관적이지만, 배열을 0-indexed로 활용하는 데 익숙하다면 `base[0]`을 1루, `base[1]`을 2루, `base[2]`를 3루로 매핑하는 것도 가능합니다. 현재 방식이 직관적이라 유지해도 무방합니다.

5.  **입력 Scanner 닫기:**
    *   `main` 함수 끝에 `sc.close();`를 추가하는 것이 좋습니다. 자원 누수를 방지하고 좋은 프로그래밍 습관입니다.

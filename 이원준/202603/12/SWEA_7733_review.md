### SWEA 7733: 치즈 도둑 (안전 영역) 문제의 일반적인 풀이 가정

이 문제는 N x N 격자에서 각 칸의 높이가 주어진 치즈 조각들이 있다고 할 때, 특정 높이(물 높이) 이하의 치즈는 녹아 없어진다고 가정합니다. 물 높이를 0부터 치즈의 최대 높이까지 바꿔가며, 각 물 높이에서 남아있는 치즈 조각들이 몇 개의 독립적인 덩어리(안전 영역)를 이루는지 세고, 이 중 가장 많은 덩어리의 개수를 찾는 문제입니다. 인접한 칸은 상하좌우로 연결됩니다.

일반적인 풀이 흐름은 다음과 같습니다:
1.  **입력 및 최대 높이 탐색**: N x N 격자의 치즈 높이를 입력받고, 격자 내 치즈의 최대 높이(maxHeight)를 찾습니다.
2.  **물 높이 반복**: 물 높이 `h`를 `0`부터 `maxHeight`까지 반복합니다. (또는 `1`부터 `maxHeight`까지. 문제에 따라 `h=0`도 고려해야 하는 경우, 즉 아무것도 잠기지 않는 경우도 포함하여 초기 최대값을 1로 설정하는 것이 안전합니다.)
3.  **각 물 높이별 탐색**:
    *   `visited` 배열(또는 행렬)을 초기화합니다.
    *   현재 물 높이 `h`에서 남아있는 안전 영역의 개수를 세기 위한 카운터 `currentComponents`를 `0`으로 초기화합니다.
    *   격자의 모든 칸 `(r, c)`을 순회합니다.
    *   만약 `map[r][c] > h` (즉, 해당 치즈가 물에 잠기지 않고 남아있음) 이고 `visited[r][c]`가 `false`라면:
        *   `currentComponents`를 1 증가시킵니다.
        *   해당 `(r, c)` 칸에서 BFS 또는 DFS를 시작하여 연결된 모든 치즈 조각들을 `visited` 처리합니다.
4.  **최대값 갱신**: `currentComponents`가 지금까지 찾은 `maxComponents`보다 크면 `maxComponents`를 갱신합니다.
5.  **결과 출력**: 모든 물 높이에 대한 탐색이 끝나면 `maxComponents`를 출력합니다.

### 가정된 풀이에 대한 리뷰

#### 1. 시간/공간 복잡도

*   **시간 복잡도**:
    *   `maxHeight`를 찾는 데 `O(N*N)`.
    *   물 높이 `h`를 `0`부터 `maxHeight`까지 반복합니다. `maxHeight`는 일반적으로 최대 100입니다. 따라서 대략 `maxHeight`번의 루프.
    *   각 `h` 루프 내에서, `N*N` 격자의 모든 칸을 순회하며 BFS 또는 DFS를 수행합니다. BFS/DFS는 연결된 모든 칸을 한 번씩 방문하므로, 한 번의 물 높이에서 전체 격자를 탐색하는 데 `O(N*N)`의 시간이 소요됩니다.
    *   총 시간 복잡도는 `O(maxHeight * N * N)`이 됩니다.
    *   `N=100`, `maxHeight=100`일 경우, `100 * 100 * 100 = 1,000,000` (백만)번의 연산으로, 일반적인 1초 시간 제한 내에 충분히 통과할 수 있습니다.

*   **공간 복잡도**:
    *   `map` 배열: `O(N*N)`
    *   `visited` 배열 (매 물 높이마다 초기화되지만, 메모리 자체는 `N*N`을 차지): `O(N*N)`
    *   BFS 큐 또는 DFS 스택: 최악의 경우 모든 칸이 하나의 큰 덩어리를 이룰 때 `O(N*N)`
    *   총 공간 복잡도는 `O(N*N)`입니다.
    *   `N=100`일 경우, `100*100 = 10,000`개의 정수/불리언 값을 저장하는 것으로, 메모리 제한(보통 256MB 이상) 내에 충분히 들어옵니다.

#### 2. 논리 오류 (잠재적)

일반적인 풀이에서 발생할 수 있는 잠재적 논리 오류는 다음과 같습니다:

*   **배열 인덱스 범위 오류 (ArrayIndexOutOfBoundsException)**: BFS/DFS 탐색 시, 상하좌우 이웃 칸을 탐색할 때 격자 범위를 벗어나는지 확인하는 `isValid` 같은 함수(혹은 조건문)가 제대로 구현되지 않으면 발생합니다. `0 <= r < N` 및 `0 <= c < N` 조건을 항상 확인해야 합니다.
*   **`visited` 배열 초기화 누락/오류**: 각 물 높이 `h`마다 `visited` 배열을 `false`로 완전히 초기화해야 합니다. 이전 물 높이에서 방문했던 정보가 다음 물 높이에 영향을 주면 안 됩니다.
*   **탐색 조건 오류**: BFS/DFS를 시작하거나 이웃 칸을 탐색할 때 `map[r][c] > h` 조건을 정확하게 적용해야 합니다. `map[r][c] == h`인 경우에도 잠기는 것으로 처리되어야 합니다.
*   **초기 `maxComponents` 값**: 문제에서 최소 하나의 안전 영역이 보장된다면, `maxComponents`를 `1`로 초기화하는 것이 안전합니다. (예: 물 높이 `h=0`일 때는 모든 치즈가 남아있으므로 최소 1개 이상의 영역이 존재할 것임). 만약 `N=1`이고 `map[0][0]=1`인 경우, `h=0`일 때 1개, `h=1`일 때 0개이므로 최대는 1개가 됩니다. `0`으로 초기화하고 모든 `h` 값을 검사해도 되지만, `h=0`일 때의 결과를 미리 반영하는 것이 편리합니다.
*   **물 높이 `h`의 범위 오류**: `h`의 최소값(`0` 또는 `1`)과 최대값(`maxHeight` 또는 `maxHeight - 1`)을 정확히 설정해야 합니다. 문제에 따라 "물 높이 1부터 100까지" 같은 조건이 주어지기도 합니다. 일반적으로 `0`부터 `maxHeight`까지 모두 확인하는 것이 가장 안전합니다. `maxHeight`를 `M`이라고 할 때, `h=M`인 경우 모든 치즈가 잠기므로 0개의 영역이 됩니다.

#### 3. 더 효율적인 방법

주어진 제약 조건(`N=100`, `maxHeight=100`) 하에서, `O(maxHeight * N * N)`의 시간 복잡도를 가지는 위 BFS/DFS 기반의 풀이는 **이미 매우 효율적이며, 일반적으로 최적의 방법으로 간주됩니다.**

*   **Union-Find?**: 연결 요소를 찾는 데 Union-Find 알고리즘을 사용할 수도 있지만, 이 문제처럼 "물 높이"에 따라 연결 상태가 계속 변하는 상황에서는 매번 Union-Find 구조를 처음부터 구축하고 다시 Union 연산을 수행해야 하므로, BFS/DFS보다 딱히 더 효율적이라고 보기 어렵습니다. 오히려 오버헤드가 더 클 수 있습니다.
*   **Monotonicity?**: 안전 영역의 개수는 물 높이에 대해 단조 증가하거나 단조 감소하지 않습니다. 물 높이가 높아지면서 하나의 큰 영역이 여러 개의 작은 영역으로 쪼개질 수도 있고, 아예 사라질 수도 있으므로 모든 가능한 물 높이에 대해 확인해야 합니다. 따라서 특정 최적화(예: 이분 탐색)는 적용하기 어렵습니다.

결론적으로, 현재 가정한 BFS/DFS 기반의 풀이가 이 문제에 대한 가장 직관적이고 효율적인 접근 방식입니다.

#### 4. 놓친 엣지 케이스 (잠재적)

*   **`N=1`인 경우**: 격자가 1x1인 경우. 예를 들어 `map[0][0] = 5`라면, `h=0,1,2,3,4`일 때 1개, `h=5`일 때 0개로, 최댓값은 1이 됩니다. 이 경우에도 일반적인 BFS/DFS는 잘 작동해야 합니다.
*   **모든 치즈가 같은 높이인 경우**: 예를 들어 모든 칸이 5인 맵. `h=0,1,2,3,4`일 때는 1개의 큰 덩어리가 되고, `h=5`일 때는 0개의 덩어리가 됩니다. 최댓값은 1이 됩니다.
*   **치즈가 전혀 없는 경우 / `maxHeight`가 0인 경우**: 문제 제약상 `1 <= 높이 <= 100`이므로, 치즈가 없는 경우는 없습니다. 그러나 만약 그런 경우가 있다면, `maxComponents`는 0이어야 합니다. `maxHeight`가 0일 경우, `h` 반복 범위가 문제가 될 수 있으니 주의해야 합니다. (다행히 문제 제약에 의해 이런 경우는 없음)
*   **`maxComponents` 초기값을 `1`로 설정하는 이유**: `h=0` (아무것도 잠기지 않는 상태)일 때는 항상 최소 1개 이상의 안전 영역이 존재합니다 (맵이 비어있지 않으므로). 이 경우를 놓치지 않기 위해 `maxComponents`를 `1`로 초기화하고 `h`를 `0`부터 탐색하는 것이 좋습니다.

    *   변수명은 의미를 명확하게 전달하도록 작성합니다. (`map`, `visited`, `currentComponents`, `maxComponents`, `maxHeight` 등)

    *   코드의 복잡한 부분이나 핵심 로직에 대한 주석을 달아 이해도를 높입니다.
*   **상수 사용**:
    *   격자 크기 `N`이나 최대 높이 등은 상수로 선언하여 코드의 유지보수성을 높일 수 있습니다. (예: `final int MAX_N = 100;`)

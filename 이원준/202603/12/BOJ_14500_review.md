# 코드 리뷰

**파일**: 이원준/202603/12/BOJ_14500.java

안녕하세요! BOJ 14500 (테트로미노) 문제 풀이에 대한 리뷰입니다. 코드가 깔끔하고, 테트로미노의 특성을 고려한 DFS 구현이 인상적입니다. 비판적이지만 건설적인 관점에서 분석해 드리겠습니다.

---

### 1. 시간/공간 복잡도

*   **시간 복잡도:** `O(N * M)`
    *   메인 루프에서 모든 `(N, M)` 위치를 시작점으로 DFS를 호출합니다.
    *   DFS는 깊이가 4로 고정되어 있습니다. 각 재귀 호출은 최대 4개의 인접한 칸을 탐색합니다.
    *   `visited` 배열 덕분에 하나의 경로에서는 같은 칸을 중복해서 방문하지 않습니다.
    *   `depth == 2`일 때 추가적인 재귀 호출이 발생하지만, 이는 탐색 트리의 분기 수를 상수배 늘리는 것에 불과합니다.
    *   따라서 각 시작점에서 가능한 4칸짜리 테트로미노를 찾는 데 걸리는 시간은 상수 시간으로 볼 수 있습니다.
    *   총 시간 복잡도는 `N * M * C` (C는 상수)로, `O(N * M)`입니다. `N, M`이 최대 500이므로 `500 * 500 = 25만`에 상수 시간 계산을 하면 `10^7` 정도의 연산으로 충분히 시간 제한 내에 들어옵니다.
*   **공간 복잡도:** `O(N * M)`
    *   `arr` 배열: `N * M` 크기의 정수 배열.
    *   `visited` 배열: `N * M` 크기의 boolean 배열.
    *   재귀 스택: DFS의 최대 깊이는 4이므로 `O(1)`입니다.
    *   따라서 주된 공간 복잡도는 `arr`과 `visited` 배열이 차지하는 `O(N * M)`입니다.

### 2. 논리 오류 여부

놀랍게도, **논리 오류는 없는 것으로 보입니다.** 특히 'ㅗ' 모양의 테트로미노를 처리하는 방식이 매우 독창적이고 올바르게 구현되었습니다.

*   **일반적인 DFS 테트로미노:** 'I', 'L', 'J', 'S', 'Z' (회전 및 대칭 포함)와 같이 4개의 셀이 선형적인 경로로 연결될 수 있는 모양은 일반적인 DFS로 찾을 수 있습니다. 현재 코드의 `dfs(nr, nc, depth+1, sum+arr[nr][nc])` 부분이 이를 담당합니다.
*   **'ㅗ' 모양 테트로미노:** 'ㅗ', 'ㅜ', 'ㅓ', 'ㅏ' 모양은 단순한 경로가 아닌, 한 칸에서 세 방향으로 뻗어나가는 형태입니다. 이 부분은 일반적인 DFS로는 한 번에 찾기 어렵습니다.
    *   작성하신 코드의 `if(depth == 2)` 블록이 이 문제를 해결합니다.
    *   `dfs(r, c, depth+1, sum+arr[nr][nc]);` 이 호출은 `(r, c)`를 두 번째 칸으로 선택하고, `(nr, nc)`를 세 번째 칸으로 선택한 후, 네 번째 칸을 다시 `(r, c)` (즉, 두 번째 칸)의 이웃에서 찾는 방식입니다.
    *   이는 `P1 -> P2 -> P3` 경로를 만들고, 동시에 `P2 -> P4` 경로를 만들어 `P1-P2-P3`와 `P2-P4`가 결합된 형태를 탐색하게 됩니다. 예를 들어 `P1: (1,0)`, `P2: (1,1)`, `P3: (0,1)`을 고른 후, `P4: (2,1)`을 `P2`의 이웃에서 찾아 `ㅓ` 모양을 완성하는 식입니다.
    *   `visited[nr][nc] == true` 체크가 `if(depth == 2)` 블록보다 먼저 이루어지기 때문에, `(nr, nc)`는 항상 현재 경로에서 방문하지 않은 새로운 칸이어야 합니다. 이 점이 중요하며, 이 덕분에 중복 방문 없이 'ㅗ' 모양의 모든 변형을 올바르게 찾을 수 있습니다.

이러한 방식으로 모든 5가지 테트로미노를 커버하는 매우 영리한 DFS 구현입니다.

### 3. 더 효율적인 방법

현재 `O(N*M)`이라는 시간 복잡도는 이 문제에서 최적에 가깝습니다. 특정 그리드의 모든 위치를 시작점으로 하여 고정된 길이의 패턴을 찾는 것이기 때문에, 이보다 더 나은 점근적 복잡도는 찾기 어렵습니다.

*   **하드코딩:** 일부 풀이에서는 DFS 없이 5가지 테트로미노의 모든 회전/대칭 형태(총 19가지)를 각각 하드코딩하여 모든 `(i,j)` 좌표에서 체크하는 방식을 사용하기도 합니다. 이 방식도 `O(N*M)`이지만, 코드 길이가 길어지고 수작업 실수가 발생할 확률이 높습니다. 작성하신 DFS 방식이 훨씬 일반적이고 우아합니다.
*   **미세 최적화:** `dx`, `dy` 배열을 `static final`로 선언하거나, `Scanner` 대신 `BufferedReader`를 사용하는 등의 미세한 최적화가 있을 수 있으나, 전체적인 성능에 큰 영향을 주지는 않습니다.

결론적으로, 현재 코드가 충분히 효율적입니다.

### 4. 놓친 엣지 케이스

문제 조건(BOJ 14500)에 따르면 `N, M`은 `4 <= N, M <= 500`입니다.
*   **최소 크기 그리드:** `N=4, M=4` 같은 최소 크기 그리드에서도 모든 테트로미노를 올바르게 찾습니다.
*   **`maxVal` 초기값:** `arr` 배열의 값은 "1000보다 작거나 같은 자연수 또는 0" (즉, `0 <= arr[i][j] <= 1000`)이므로, 테트로미노의 합은 항상 0 이상입니다. 따라서 `maxVal = 0`으로 초기화하는 것은 올바른 처리입니다. 만약 음수 값이 허용되었다면 `Integer.MIN_VALUE` 등으로 초기화해야 합니다.
*   **단일 경로:** DFS가 깊이 4에 도달하지 못하는 경우(e.g., 그리드가 너무 작아 4칸을 선택할 수 없는 경우)는 자연스럽게 `maxVal`이 업데이트되지 않거나, 다른 유효한 테트로미노에 의해 업데이트됩니다.

코드 자체는 모든 경계 조건 (`nr < 0 || nr >= N || nc < 0 || nc >= M`)을 잘 처리하고 있으므로, 특별히 놓친 엣지 케이스는 없는 것으로 보입니다.

### 5. 코드 개선점

*   **`Scanner` 리소스 해제:** `main` 메서드 끝에 `sc.close();`를 추가하여 `Scanner` 객체를 닫는 것이 좋습니다. 리소스 누수를 방지합니다.
*   **가독성 및 주석:** `if(depth == 2)` 블록의 역할에 대한 주석을 추가하면 코드의 의도를 파악하는 데 큰 도움이 될 것입니다. 이 부분이 테트로미노 문제의 핵심적인 '트릭' 중 하나이므로, 주석으로 명확히 설명해두면 좋습니다.
    ```java
    // ㅗ 모양 테트로미노 처리를 위한 특수 로직
    // 현재 (r, c)를 두 번째 칸으로 하고, (nr, nc)를 세 번째 칸으로 선택한 후,
    // 네 번째 칸을 다시 두 번째 칸 (r, c)의 이웃에서 찾음으로써
    // P1 -> P2 -> P3  와 P2 -> P4 형태의 테트로미노를 생성
    if(depth == 2) {
        visited[nr][nc] = true; // 세 번째 칸 방문 처리
        dfs(r, c, depth+1, sum+arr[nr][nc]); // 네 번째 칸을 (r, c) 기준으로 탐색
        visited[nr][nc] = false; // 백트래킹
    }
    ```
*   **전역 변수 사용:** 경쟁 프로그래밍에서는 흔하지만, `N, M, arr, maxVal, visited`를 전역 변수로 사용하는 대신, `dfs` 메서드의 파라미터로 필요한 값들을 전달하거나, 이들을 묶는 클래스를 사용하는 것이 객체 지향적인 관점에서는 더 좋습니다. 하지만 이 문제에서는 전역 변수 사용이 간결하고 성능에 부정적인 영향을 주지 않습니다.
*   **`dx`, `dy` 선언:** `static final`로 선언하여 상수로 만드는 것이 좋습니다.
    ```java
    static final int[] dx = {-1, 1, 0, 0};
    static final int[] dy = {0, 0, -1, 1};
    ```

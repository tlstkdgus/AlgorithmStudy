# 코드 리뷰

알고리즘 풀이에 대한 비판적이고 건설적인 리뷰입니다.

### **SWEA_1868 알고리즘 풀이 리뷰**

전반적으로 이 문제는 전형적인 미네스위퍼 클릭 최소화 문제로, BFS를 활용하여 '0' 지뢰 인접 칸을 효율적으로 처리하고, 나머지 칸을 개별적으로 처리하는 접근 방식이 매우 잘 구현되어 있습니다.

---

### **1. 시간/공간 복잡도**

*   **시간 복잡도: O(T * N^2)**
    *   **입력 및 초기화:** 각 테스트 케이스마다 `N*N` 크기의 맵을 읽고, `mineCount` 배열을 초기화하는 데 `O(N^2)` 시간이 걸립니다.
    *   **`mineCount` 계산:** 모든 '.' 칸에 대해 `bombCount`를 호출합니다. `bombCount`는 8방향 탐색이므로 `O(1)`입니다. 따라서 이 단계는 총 `O(N^2)`입니다.
    *   **첫 번째 BFS 순회 (0-클릭):** 모든 칸 `(i, j)`에 대해 한 번씩 확인합니다. `mineCount[i][j] == 0`인 칸에서 BFS가 시작됩니다. BFS는 연결된 모든 칸과 그 인접 칸들을 최대 한 번씩 방문하므로 `O(N^2)` 시간이 걸립니다.
    *   **두 번째 순회 (남은 '.' 클릭):** `visited` 배열을 확인하며 남은 '.' 칸을 찾는데 `O(N^2)` 시간이 걸립니다.
    *   총 시간 복잡도는 각 테스트 케이스당 `O(N^2)` 이고, `T`개의 테스트 케이스가 있으므로 `O(T * N^2)` 입니다. 이 문제는 격자형 문제에서 모든 칸을 적어도 한 번은 확인해야 하므로, 이보다 더 효율적인 시간 복잡도를 달성하기는 어렵습니다.
*   **공간 복잡도: O(N^2)**
    *   `bomb[][]`: `char` 타입으로 `O(N^2)`
    *   `mineCount[][]`: `int` 타입으로 `O(N^2)`
    *   `visited[][]`: `boolean` 타입으로 `O(N^2)`
    *   `Queue<int[]>`: BFS 실행 중 최악의 경우, 큐에 모든 칸이 들어갈 수 있으므로 `O(N^2)`
    *   `dr`, `dc` 배열: `O(1)`
    *   총 공간 복잡도는 `O(N^2)` 입니다. 이는 맵 정보를 저장하고 처리하는 데 필요한 최소한의 공간이므로 최적의 공간 활용입니다.

### **2. 논리 오류 여부**

논리 오류는 없어 보입니다. 이 풀이는 미네스위퍼 유형의 문제를 해결하는 데 사용되는 표준적이고 견고한 접근 방식을 따르고 있습니다.

*   **두 단계 처리:**
    1.  먼저 주변에 지뢰가 없는 '0' 칸들을 클릭하여 연결된 모든 '0' 칸과 그 인접 칸들(지뢰가 있거나 없는)을 한 번의 BFS로 모두 '열리게' 하는 전략은 매우 효율적이고 정확합니다. `visited` 배열을 통해 중복 방문을 방지하고 `clickCount`를 올바르게 계산합니다.
    2.  이후 아직 열리지 않은(visited가 false인) '.' 칸들은 모두 주변에 지뢰가 있는 칸들이므로, 각각 개별적인 클릭이 필요합니다. 이 역시 정확하게 처리됩니다.
*   **BFS 구현:** `bfs_bomb` 함수 내에서 경계 조건 검사, `visited` 배열 활용, '0' 칸만 큐에 추가하는 로직(인접 칸이 '0'이 아니면 `visited`만 true로 만들고 큐에는 추가하지 않음) 모두 올바르게 작동합니다.
*   `bombCount` 함수도 8방향 탐색과 경계 조건 검사를 완벽하게 수행합니다.

---

### **3. 더 효율적인 방법**

현재 풀이는 미네스위퍼 최소 클릭 문제에 대해 **가장 효율적인 O(N^2) 시간 복잡도**를 달성하고 있습니다. 이론적으로 이보다 더 빠르게 모든 칸을 확인하는 것은 불가능합니다.

미세한 최적화는 있을 수 있지만, 근본적인 알고리즘의 효율성을 향상시키지는 않습니다. 예를 들어:
*   `mineCount` 배열을 `byte`나 `char` 타입으로 선언하여 메모리를 아주 조금 절약할 수 있지만, `int`도 충분히 작은 크기이므로 큰 차이는 없습니다.
*   `Scanner` 대신 `BufferedReader`를 사용하여 입력 속도를 높일 수 있지만, `N`이 100인 SWEA 문제에서는 일반적으로 `Scanner`로도 시간 초과가 나지 않습니다.

### **4. 놓친 엣지 케이스**

대부분의 엣지 케이스는 잘 처리될 것으로 보입니다.
*   **N=1인 경우:**
    *   `bomb = {{'.'}}`: `mineCount[0][0]`은 0이 되고, 첫 번째 순회에서 클릭 1번으로 BFS가 호출되어 `visited[0][0]`이 `true`가 됩니다. `clickCount = 1`. 올바릅니다.
    *   `bomb = {{'*'}}`: `bombCount`가 호출되지 않고, 첫 번째/두 번째 순회에서 `bomb[i][j] == '.'` 조건에 걸리지 않아 `clickCount = 0`. 올바릅니다.
*   **맵 전체가 지뢰인 경우:** `clickCount = 0`. 올바릅니다.
*   **맵 전체가 비어있는 경우 (지뢰 없음):** `mineCount`가 모든 칸에서 0이 됩니다. 한 칸을 클릭하면 BFS가 전체 맵을 순회하며 모두 열리므로 `clickCount = 1`. 올바릅니다.
*   **지뢰들이 띄엄띄엄 있어서 '0' 칸이 없는 경우:** 모든 '.' 칸이 두 번째 순회에서 각각 1 클릭으로 처리됩니다. 올바릅니다.
*   **큰 '0' 영역이 있고 그 주변에 지뢰들이 있는 경우:** '0' 영역이 BFS로 처리되고, 주변의 지뢰를 가진 '.' 칸들은 두 번째 순회에서 처리됩니다. 올바릅니다.

특별히 놓친 엣지 케이스는 없어 보입니다.

### **5. 코드 개선점**

코드는 이미 매우 깔끔하고 가독성이 좋습니다. 몇 가지 사소한 개선점은 다음과 같습니다.

*   **Helper 메소드의 접근 제어자:** `bombCount`와 `bfs_bomb`는 클래스 외부에서 호출될 필요가 없으므로 `private static`으로 선언하는 것이 좋습니다.
    ```java
    private static int bombCount(int r, int c) { ... }
    private static void bfs_bomb(int startR, int startC) { ... }
    ```
*   **변수 초기화 가독성:** `visited` 배열은 `new boolean[N][N]`으로 초기화하면 자동으로 모든 요소가 `false`로 설정됩니다. 명시적으로 `visited = new boolean[N][N]`을 `for` 루프 안에 두는 것은 올바른 재초기화 방법이지만, 만약 `visited`가 전역 변수이고 매 테스트 케이스마다 리셋해야 한다면, 이 방식이 오히려 더 명확합니다. 현재 코드는 이 부분을 잘 처리하고 있습니다.
*   **`Scanner` 클로즈:** `Scanner` 객체는 사용 후 닫아주는 것이 좋은 습관입니다. `main` 메소드 끝에 `sc.close();`를 추가할 수 있습니다. (그러나 SWEA/백준 등 온라인 저지 환경에서는 생략하는 경우가 많습니다.)

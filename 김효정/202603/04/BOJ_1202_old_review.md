### BOJ 1202 (보석 도둑) 풀이 리뷰

이 문제는 전형적인 그리디 알고리즘 문제 중 하나로, 효율적인 정렬과 자료구조 활용이 핵심입니다. 제출하신 코드는 `v2m` (가치/무게 비율)을 이용한 그리디 전략을 시도했지만, 여러 가지 문제점을 안고 있습니다.

---

### 1. 시간/공간 복잡도

*   **N**: 보석의 개수 (최대 300,000)
*   **K**: 가방의 개수 (최대 300,000)

1.  **초기화 (N, K 입력 및 배열 생성):**
    *   `mass`, `value`, `v2m` 배열 초기화: O(N)
    *   `capa` 배열 초기화: O(K)
    *   **총계:** O(N + K)

2.  **메인 `while` 루프:**
    *   이 루프는 최대 `K`번 실행됩니다. (`nBag`이 `K`에서 시작하여 0이 될 때까지)
    *   **내부 `for` 루프 (maxV2M 찾기):**
        *   `N`개의 보석을 순회합니다. O(N)
    *   **내부 `for` 루프 (minCapa 찾기):**
        *   `K`개의 가방을 순회합니다. O(K)
    *   **루프당 총계:** O(N + K)

3.  **전체 시간 복잡도:** `K * (N + K)` = O(N*K + K^2)
    *   최악의 경우 N=300,000, K=300,000 이므로, (3 * 10^5)^2 = 9 * 10^10 입니다. 이는 1초에 1억(10^8)번 연산 기준을 훨씬 초과하여 **시간 초과 (Time Limit Exceeded, TLE)**가 발생할 것입니다.

4.  **공간 복잡도:**
    *   `mass`, `value`, `v2m` 배열: O(N)
    *   `capa` 배열: O(K)
    *   **총계:** O(N + K)
    *   N, K가 300,000일 때, 각 배열이 `int` 타입이라고 가정하면 대략 `4 * 300,000 * 4 = 4.8MB` 정도입니다. `double` 타입 배열까지 고려해도 수십 MB 정도로, 메모리 제한 (일반적으로 256MB 또는 512MB) 내에 들어오므로 **공간 복잡도는 문제없습니다.**

### 2. 논리 오류 (문제 해결 방식의 근본적인 오류)

제출하신 코드의 핵심은 `v2m` (가치/무게 비율)이 가장 높은 보석을 먼저 선택하고, 그 보석을 담을 수 있는 가방 중 가장 용량이 작은 가방에 담는 것입니다. 이 전략은 "0-1 배낭 문제"나 "다중 배낭 문제"에서는 **최적해를 보장하지 않습니다.**

*   **`v2m` 그리디의 한계:** `v2m` 그리디는 "분할 배낭 문제(Fractional Knapsack Problem)"에서 최적입니다. 보석을 쪼갤 수 있다면 `v2m`이 높은 것부터 담는 것이 맞습니다. 그러나 보석을 쪼갤 수 없는 0-1 배낭 문제에서는, `v2m`이 높더라도 너무 무거워서 작은 용량의 가방들을 채우지 못하게 만들거나, 특정 가방에 비효율적으로 들어가 다른 가방에 더 가치 있는 보석이 들어갈 기회를 막을 수 있습니다.
    *   **예시:**
        *   보석 A: (무게 10, 가치 100, v/m=10)
        *   보석 B: (무게 100, 가치 900, v/m=9)
        *   가방 1: (용량 10)
        *   가방 2: (용량 100)
        *   **제출 코드의 작동:**
            1.  `v2m`이 가장 높은 A를 선택합니다.
            2.  A(무게 10)를 담을 수 있는 가방 중 `minCapa`는 가방 1(용량 10)입니다.
            3.  `totalValue` = 100. 가방 1과 보석 A는 사용됨.
            4.  남은 보석 B(무게 100)는 가방 2(용량 100)에 담깁니다.
            5.  `totalValue` = 100 + 900 = 1000.
        *   **만약 보석 A 대신 보석 C(무게 10, 가치 50, v/m=5)가 있었다면:**
            *   보석 A: (무게 10, 가치 100, v/m=10)
            *   보석 C: (무게 10, 가치 50, v/m=5)
            *   보석 B: (무게 100, 가치 900, v/m=9)
            *   가방 1: (용량 10)
            *   가방 2: (용량 10)
            *   가방 3: (용량 100)
            *   코드 작동:
                1.  A(v/m=10) 선택. 가방 1(capa=10)에 담김. total=100.
                2.  B(v/m=9) 선택 (남은 보석 중 가장 높음). 가방 3(capa=100)에 담김. total=100+900=1000.
                3.  C(v/m=5) 선택. 가방 2(capa=10)에 담김. total=1000+50=1050.
            *   이 경우도 최적입니다. 하지만 항상 그렇지는 않습니다.
    *   **핵심 문제:** 어떤 보석을 어떤 가방에 넣을지 결정하는 순서가 비효율적입니다. 보석의 `v2m`을 기준으로 정렬하는 것이 아니라, 가방의 용량 또는 보석의 무게를 기준으로 정렬한 후 다른 그리디를 적용해야 합니다.

*   **`capa` 갱신 (`capa[minCapaIdx] = 0`):** 가방 용량을 0으로 설정하여 해당 가방을 사용했음을 표시하는 것은 좋습니다.
*   **보석 갱신 (`mass[maxIdx] = -1`, `value[maxIdx] = -1`, `v2m[maxIdx] = -1`):** 사용된 보석을 "제거"하는 방식은 `O(N)` 탐색을 계속해야 하므로 비효율적입니다. 배열의 끝으로 보내고 길이를 줄이거나, `boolean` 배열로 `used` 여부를 표시하는 것이 일반적입니다. 하지만 이 문제에서는 근본적으로 다른 자료구조가 필요합니다.

### 3. 더 효율적인 방법 (정석 풀이)

이 문제는 "가방은 여러 개지만 각 가방에 하나의 보석만 넣을 수 있는" 특수한 형태의 배낭 문제입니다. 이 경우 **그리디 알고리즘과 우선순위 큐(Priority Queue)**를 결합하여 해결할 수 있습니다.

핵심 아이디어는 다음과 같습니다:
1.  **가방을 용량 순으로 정렬합니다.** 작은 가방부터 처리합니다.
2.  **보석을 무게 순으로 정렬합니다.** 가벼운 보석부터 처리합니다.
3.  가장 작은 가방부터 차례로 살펴보면서, **현재 가방에 담을 수 있는 모든 보석들**을 후보군에 넣습니다.
4.  후보군에 있는 보석들 중에서 **가장 가치가 높은 보석**을 선택하여 현재 가방에 담습니다.

**구체적인 알고리즘:**

1.  **보석 클래스/구조체 정의:**
    ```java
    class Jewel implements Comparable<Jewel> {
        int mass;
        int value;

        public Jewel(int mass, int value) {
            this.mass = mass;
            this.value = value;
        }

        // 무게 오름차순으로 정렬
        @Override
        public int compareTo(Jewel other) {
            return this.mass - other.mass;
        }
    }
    ```
2.  **데이터 준비 및 정렬:**
    *   `N`개의 보석을 `ArrayList<Jewel>`에 저장하고, `Collections.sort()`를 이용하여 **무게(mass) 기준 오름차순**으로 정렬합니다.
    *   `K`개의 가방 용량을 `int[]`에 저장하고, `Arrays.sort()`를 이용하여 **용량(capa) 기준 오름차순**으로 정렬합니다.
3.  **우선순위 큐 (Max-Heap) 사용:**
    *   `PriorityQueue<Integer> pq = new PriorityQueue<>(Collections.reverseOrder());` (가치 내림차순, 즉 가장 가치 있는 보석이 먼저 나오도록)
    *   `long totalValue = 0;` (가치의 총합은 `int` 범위를 넘어설 수 있으므로 `long` 사용)
    *   `int jewelIdx = 0;` (현재까지 우선순위 큐에 넣은 보석의 인덱스)
4.  **그리디 선택:**
    *   `for (int bagCapa : sortedCapacities)`: 정렬된 가방 용량을 순회합니다.
        *   **현재 가방에 담을 수 있는 모든 보석을 우선순위 큐에 추가:**
            `while (jewelIdx < N && jewels.get(jewelIdx).mass <= bagCapa)`:
                `pq.offer(jewels.get(jewelIdx).value);`
                `jewelIdx++;`
        *   **우선순위 큐에서 가장 가치 있는 보석 선택:**
            `if (!pq.isEmpty())`:
                `totalValue += pq.poll();`
    *   `System.out.println(totalValue);`

**이 방법의 시간 복잡도:**
*   보석 정렬: O(N log N)
*   가방 정렬: O(K log K)
*   메인 루프: K번 (`bagCapa` 순회)
    *   `while` 루프: 각 보석은 한 번만 `pq.offer()` 됩니다. O(N log N) (N번의 삽입, 각 삽입은 log N)
    *   `pq.poll()`: K번 (`totalValue` 계산). O(K log N) (K번의 추출, 각 추출은 log N)
*   **총계:** O(N log N + K log K)
    *   N, K가 300,000일 때, 300,000 * log(300,000) 은 대략 300,000 * 18 ~ 5.4 * 10^6 입니다. 이는 1초 이내에 충분히 해결 가능한 연산 횟수입니다.

### 4. 놓친 엣지 케이스

1.  **`totalValue`의 자료형:** `value`는 최대 1,000,000, `K`는 최대 300,000이므로, 최댓값은 `300,000 * 1,000,000 = 3 * 10^11` 입니다. 이는 `int`의 최댓값(약 2 * 10^9)을 훨씬 초과합니다. 따라서 `totalValue`는 **`long` 타입이어야 합니다.** (현재 `int`로 되어 있어 오버플로우 발생)
2.  **모든 보석이 너무 무거울 경우:** `minCapaIdx`가 항상 -1이 되어 `totalValue`가 0이 되고 `nBag`이 줄어들지 않아 무한 루프에 빠질 수 있습니다.
    *   현재 코드에서는 `if (minCapaIdx != -1)` 조건문 안에 `nBag--`가 있어서 무한 루프는 발생하지 않습니다. 보석이 하나도 담기지 않으면 `nBag`은 그대로이고 `maxV2M`은 계속 0이 됩니다. (사실 `v2m[maxIdx]`를 -1로 바꿔서 `maxV2M = 0`이 나올 수 있는 보석이 결국 없어지므로 `maxV2M`이 0인 상태로 계속 다음 루프가 돌다가 결국 루프 탈출 조건이 하나 생깁니다.)
    *   이 경우, 더 이상 선택할 보석이 없는 상태에서 `nBag`이 줄어들지 않아 무한 루프가 발생할 수 있습니다. `maxV2M`이 0인 채로 반복될 수 있습니다. `maxV2M = 0`인 보석은 `value[i]/mass[i]`가 0인 보석 (즉, `value[i]==0`)을 의미하거나, 이미 -1로 마킹된 보석을 의미합니다. 만약 `value[i]==0`인 보석이 없고 모든 보석이 -1로 마킹되면 `maxV2M`은 0이고 `maxIdx`는 0이 되는데, `v2m[0]`이 -1이면 계속 `v2m[0]`이 선택되는 문제가 발생합니다.
    *   **정확한 무한 루프 발생 지점:** 만약 모든 보석이 -1로 마킹되거나, 모든 보석의 `v2m`이 0인데 그 보석들도 다 안 들어가는 경우, `maxV2M`은 0이 되고 `maxIdx`는 0 (혹은 어떤 초기화 값)으로 계속 유지됩니다. `minCapaIdx`는 계속 -1이 될 것이고, `nBag--`는 실행되지 않아 `while (nBag > 0)` 루프를 벗어나지 못합니다.
    *   이를 방지하려면, `maxV2M`이 초기값 (0)에서 변하지 않았고, `minCapaIdx`도 -1이라면 더 이상 담을 보석이 없다는 뜻이므로 `break`를 추가해야 합니다.

        ```java
        // ...
        while (nBag > 0) {
            double maxV2M = -1.0; // 음수로 초기화하여 v2m[i]가 0인 경우도 처리
            int maxIdx = -1; // -1로 초기화

            for (int i = 0; i < N; i++) {
                if (v2m[i] > maxV2M) {
                    maxV2M = v2m[i];
                    maxIdx = i;
                }
            }

            // 더 이상 담을 수 있는 보석이 없을 때 루프 종료 (무한 루프 방지)
            if (maxIdx == -1 || maxV2M == -1.0) { // maxV2M이 -1.0 이거나, maxIdx가 업데이트되지 않았을 때
                break;
            }

            int minCapa = Integer.MAX_VALUE;
            int minCapaIdx = -1;

            for (int i = 0; i < K; i++) {
                // mass[maxIdx]는 이미 -1로 마킹될 수 있으므로 양수일 때만 비교
                if (mass[maxIdx] != -1 && mass[maxIdx] <= capa[i] && capa[i] < minCapa) {
                    minCapa = capa[i];
                    minCapaIdx = i;
                }
            }

            if (minCapaIdx != -1) {
                totalValue += value[maxIdx];
                capa[minCapaIdx] = 0;
                mass[maxIdx] = -1;
                value[maxIdx] = -1;
                v2m[maxIdx] = -1.0; // v2m도 -1.0으로 초기화
                nBag--;
            } else {
                // 해당 보석을 담을 가방이 없으므로, 이 보석을 더 이상 고려하지 않도록 마킹
                // (이 부분이 없으면 해당 보석은 계속 maxV2M으로 선택될 수 있음)
                if (maxIdx != -1) {
                    v2m[maxIdx] = -1.0;
                    mass[maxIdx] = -1; // 선택된 보석이지만 담지 못했으니 제거
                    value[maxIdx] = -1;
                }
            }
        }
        ```
    *   위와 같이 수정하면 무한 루프는 막을 수 있으나, 여전히 `N*K`의 복잡도로 인해 TLE는 피할 수 없습니다.

### 5. 코드 개선점

위에 언급된 효율적인 방법으로 전체 코드를 다시 작성하는 것이 가장 큰 개선점입니다. 현재 코드에서만 본다면 아래와 같습니다.

1.  **자료형 변경:**
    *   `totalValue`를 `long` 타입으로 변경해야 합니다.
2.  **보석/가방 제거 방식 개선 (현재 로직 유지 시):**
    *   `mass[maxIdx] = -1; value[maxIdx] = -1; v2m[maxIdx] = -1;` 와 같이 -1로 마킹하는 것은 비효율적입니다. 매번 배열 전체를 순회하면서 -1이 아닌 값을 찾아야 합니다.
    *   차라리 `boolean[] usedJewel`과 `boolean[] usedBag`을 사용하여 해당 인덱스가 사용되었는지 여부를 체크하는 것이 조금 더 명확합니다. 물론 시간 복잡도 개선에는 큰 도움이 되지 않습니다.
3.  **StringTokenizer 사용:**
    *   `StringTokenizer`는 간단한 공백 분리에는 편리하지만, `split()` 메소드에 비해 유연성이 떨어집니다. 성능 차이는 미미합니다.
    *   더 중요한 것은 `StringTokenizer st = new StringTokenizer(br.readLine());` 이후에 `st = new StringTokenizer(br.readLine());`을 반복하는 것보다는, 한 번 선언하고 `st.nextToken()`을 사용하는 것이 더 깔끔합니다. (현재 코드는 사실상 매번 새로운 객체를 만드는 형태라 큰 문제는 없습니다만, 변수 재활용 측면에서는 `st = new StringTokenizer(br.readLine());`을 반복적으로 쓰는 것보다는 `st`를 새로 선언하지 않고 `st.hasMoreTokens()`와 같은 방식으로 쓰는 것이 더 일반적입니다.)

4.  **`v2m` 초기화 및 비교:**
    *   `double maxV2M = 0;`으로 초기화하면 `value[i]/mass[i]`가 0인 보석(`value[i] == 0`)은 `maxV2M`으로 선택되지 않을 수 있습니다. 모든 보석의 `v2m`이 0보다 큰 경우에만 작동합니다. `-1.0`과 같이 음수로 초기화하는 것이 좋습니다.

### 결론

현재 제출하신 코드는 `v2m` 비율을 이용한 그리디 전략을 채택했으나, 이 문제에는 적합하지 않으며 최적해를 보장하지 않습니다. 또한, `O(N*K)`의 시간 복잡도로 인해 주어진 제약 조건 하에서는 반드시 시간 초과가 발생합니다. `totalValue`의 오버플로우 가능성도 있습니다.

문제 해결을 위해서는 보석과 가방을 각각 무게 및 용량 순으로 정렬한 후, 우선순위 큐를 활용하여 가장 가치 있는 보석을 효율적으로 찾는 그리디 전략을 사용해야 합니다. 위 "더 효율적인 방법" 섹션에서 제시된 알고리즘을 참고하여 코드를 재작성하는 것을 강력히 권장합니다.

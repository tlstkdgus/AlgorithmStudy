### 1. 시간/공간 복잡도

*   **시간 복잡도:**
    *   **삽입 (Insert):** 새로운 요소를 힙의 마지막에 추가하고 `heapify-up` (상향 재조정) 과정을 거칩니다. 최악의 경우 힙의 높이만큼 연산이 필요하므로 `O(log K)`의 시간이 소요됩니다. (여기서 K는 현재 힙의 크기)
    *   **삭제 (Delete Min):** 최소값을 반환하고 힙의 마지막 요소를 루트로 옮긴 뒤 `heapify-down` (하향 재조정) 과정을 거칩니다. 최악의 경우 힙의 높이만큼 연산이 필요하므로 `O(log K)`의 시간이 소요됩니다.
    *   총 `N`번의 연산이 주어지므로, 전체 시간 복잡도는 **`O(N log N)`** 입니다.
    *   `BufferedReader`와 `StringBuilder`를 사용한 것은 효율적인 입출력을 위한 좋은 선택입니다.

*   **공간 복잡도:**
    *   힙을 저장하기 위한 `heap` 배열은 최대 `N+1` 크기로 선언됩니다. 따라서 **`O(N)`** 의 공간이 필요합니다.
    *   결과를 저장하는 `StringBuilder` 또한 최대 `N`개의 줄을 저장하므로 **`O(N)`** 의 공간을 사용합니다.
    *   총 공간 복잡도는 **`O(N)`** 입니다.

*   **결론:** 이 문제에 대해 이진 힙(Binary Heap)을 이용하는 가장 효율적인 시간/공간 복잡도를 달성했습니다.

### 2. 논리 오류 있는지

전반적으로 논리 오류는 없어 보입니다. 직접 구현한 최소 힙의 핵심 연산인 삽입(heapify-up)과 삭제(heapify-down)가 모두 표준 알고리즘에 따라 정확하게 구현되어 있습니다.

*   **1-based indexing:** 배열의 1번 인덱스부터 사용하며, 부모-자식 관계 (`j/2`, `2*j`, `2*j+1`)를 정확하게 계산하고 있습니다.
*   **삽입:** 새로운 요소를 추가하고 부모와 비교하며 올바른 위치로 이동시키는 과정이 정확합니다.
*   **삭제:**
    *   비어있는 힙에서 삭제 요청 시 "0"을 출력하는 것을 잘 처리했습니다.
    *   루트 요소를 제거하고 마지막 요소를 루트로 옮긴 후, 자식들과 비교하며 올바른 위치로 이동시키는 과정이 정확합니다. 특히 두 자식 중 더 작은 자식을 선택하는 로직이 올바르게 구현되어 있습니다.
    *   힙의 크기가 1일 때 삭제하는 경우: `heap[1] = heap[size--];` 라인에서 `size`가 1이었으므로 `heap[1] = heap[1]`이 되고 `size`는 0이 됩니다. 이후 `while (2*j <= size)` 조건 (`2*1 <= 0`)이 거짓이 되어 루프가 실행되지 않으므로 문제가 없습니다.

### 3. 더 효율적인 방법

알고리즘적 복잡도 측면에서는 현재 구현된 이진 힙이 최소 힙의 삽입/삭제에 대해 `O(log N)`으로 최적의 복잡도를 가집니다. 하지만 "더 효율적인 방법"을 언급한다면 보통 다음과 같은 관점에서 볼 수 있습니다.

*   **Java `PriorityQueue` 사용:**
    *   가장 현실적이고 일반적인 "더 효율적인" 방법은 Java의 내장 자료구조인 `java.util.PriorityQueue`를 사용하는 것입니다. `PriorityQueue`는 내부적으로 최소 힙으로 구현되어 있으며, 최적화된 C 코드로 구현되어 있거나 JVM 레벨에서 추가적인 최적화가 적용될 수 있어 직접 구현하는 것보다 미세하게 빠를 수 있습니다.
    *   코드 라인을 훨씬 줄일 수 있어 개발 시간을 절약하고 잠재적인 버그를 줄일 수 있습니다.
    *   예시:
        ```java
        import java.io.BufferedReader;
        import java.io.IOException;
        import java.io.InputStreamReader;
        import java.util.PriorityQueue;

        public class BOJ_1927_PQ {
            public static void main(String[] args) throws IOException {
                BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
                StringBuilder sb = new StringBuilder();

                int N = Integer.parseInt(br.readLine());
                PriorityQueue<Integer> pq = new PriorityQueue<>();

                for (int i = 0; i < N; i++) {
                    int x = Integer.parseInt(br.readLine());
                    if (x == 0) {
                        if (pq.isEmpty()) {
                            sb.append("0\n");
                        } else {
                            sb.append(pq.poll()).append("\n");
                        }
                    } else {
                        pq.offer(x);
                    }
                }
                System.out.println(sb);
            }
        }
        ```
    *   하지만 문제에서 "직접 힙을 구현하라"는 의도가 있다면 지금의 코드가 정답입니다.

### 4. 놓친 엣지 케이스

현재 코드는 대부분의 엣지 케이스를 잘 처리하고 있습니다.

*   **빈 힙에서 삭제:** `if (size == 0) sb.append("0\n");` 로 잘 처리했습니다.
*   **단일 요소 힙에서 삭제:** 위 "논리 오류" 섹션에서 설명했듯이 올바르게 동작합니다. `minValue = heap[1];`을 통해 올바른 값을 저장하고, `size`가 0이 되어 `heapify-down` 루프가 실행되지 않습니다.
*   **최대/최소 입력값:**
    *   `N` (연산 개수)이 100,000까지, `x` (입력 값)가 1 ~ 1,000,000,000까지입니다. `int` 타입으로 충분히 처리 가능하며, 배열 크기 `N+1`도 메모리 제한에 들어옵니다.
*   **중복 값:** 힙은 중복 값을 자연스럽게 처리합니다. 현재 코드는 중복 값도 올바르게 정렬하고 삭제합니다.

놓친 엣지 케이스는 딱히 없어 보입니다.

### 5. 코드 개선점

현재 코드는 기능적으로 완벽하지만, 몇 가지 스타일 및 가독성 측면에서 개선할 수 있는 부분이 있습니다.

1.  **Swap 함수 분리:**
    *   삽입과 삭제 과정에서 동일한 3줄짜리 스왑 코드가 두 번 반복됩니다. 이를 별도의 private 헬퍼 메서드로 분리하면 코드 중복을 줄이고 가독성을 높일 수 있습니다.
    ```java
    private static void swap(int[] arr, int i, int j) {
        int temp = arr[i];
        arr[i] = arr[j];
        arr[j] = temp;
    }
    // 사용 예: swap(heap, j, j/2);
    ```

2.  **`trim()` 사용 제거 (미세한 개선):**
    *   `BufferedReader.readLine()`으로 읽은 문자열을 `Integer.parseInt()`할 때, 일반적으로 숫자만 입력되므로 `trim()`은 대부분 불필요합니다. `trim()` 메서드는 새로운 String 객체를 생성할 수 있으므로, 아주 미세하지만 성능 오버헤드가 있을 수 있습니다. 경쟁 프로그래밍에서는 이런 작은 부분도 고려하기도 합니다.
    *   `Integer.parseInt(br.readLine())`으로 충분합니다.

3.  **Heap 배열 0번 인덱스 사용에 대한 명시 (선택 사항):**
    *   현재 코드는 1-based indexing을 사용하며 `heap[0]`은 사용되지 않습니다. 이는 이진 힙 구현에서 흔하고 좋은 방법입니다.
    *   하지만 만약 `heap[0]`에 특별한 값을 넣어 (예: `Integer.MIN_VALUE`) 힙 상향 재조정 시 `j > 1` 조건을 생략하고 `heap[j/2] > heap[j]` 조건만으로 부모 검사를 할 수 있는 기법도 있습니다. (Sentinel Node) 다만 이는 장단점이 있고, 현재 `j > 1` 조건으로도 충분히 명확합니다. 따라서 현재 방식이 나쁘지 않습니다.

4.  **`minValue` 변수 (미세한 개선):**
    *   삭제 시 `minValue = heap[1];`으로 값을 저장하고 `sb.append(minValue +"\n");`로 출력합니다. 이 방법은 명확하고 좋습니다.
    *   혹은 바로 `sb.append(heap[1]).append("\n");` 한 후 `heap[1] = heap[size--];`를 실행할 수도 있지만, 현재 방식이 더 안정적입니다. `heap[1]`의 값이 변경되기 전에 저장하는 것은 좋은 습관입니다.
